01 Foundations of Fine-Tuning

1.1 What Is Fine-Tuning?

Fine-tuning takes a pre-trained language model — one that has already learned general linguistic representations from a massive corpus — and adapts its parameters on a smaller, task-specific or domain-specific dataset. The foundational intuition is that pre-training captures broad knowledge about language structure, semantics, and world facts, while fine-tuning specializes the model's probability distribution to favor particular behaviors, formats, or domains.

Think of pre-training as giving the model a broad liberal arts education, and fine-tuning as specialized professional training. The model already "knows" how language works; fine-tuning teaches it how to apply that knowledge to your specific needs.

⚡ Formal Definition

Given a pre-trained model with parameters θ₀ learned by minimizing a pre-training objective Lpretrain over corpus Dpretrain, fine-tuning finds updated parameters θ* by:

θ* = arg minθ Lft(Dft; θ) where θ is initialized at θ₀

The critical insight is that θ₀ provides a strong initialization in parameter space. Without this, optimizing Lft from random weights would require orders of magnitude more data and compute. The pre-trained initialization positions the model in a region of the loss landscape where gradient descent can efficiently find a good task-specific solution — this is why fine-tuning works with as few as 1,000 examples while pre-training requires trillions of tokens.

🧮 Derive It Yourself — The Fine-Tuning Objective

If an interviewer asks you to write this formula on a whiteboard, here's how to build it from scratch:

1

Start from what you know: All supervised learning minimizes a loss over data. Write the generic form: θ* = arg minθ L(D; θ). This is just "find the parameters that minimize loss on the data."

2

Add the key constraint: Fine-tuning is not training from scratch. The only thing that makes it "fine-tuning" rather than "training" is the initialization. So annotate: "where θ is initialized at θ₀" (the pre-trained weights). That's the entire formal definition.

3

Specialize the loss: For language models, L is the cross-entropy over token predictions. For SFT, it's computed only over response tokens. For continual pre-training, it's the standard next-token loss over all tokens. The choice of Lft is what differentiates types of fine-tuning.

Interview mnemonic: Fine-tuning = standard optimization + special initialization. That's all. Everything else (loss function choice, which parameters to freeze, regularization) is a design decision layered on top.

1.2 Pre-Training vs. Fine-Tuning: The Loss Functions

Understanding the difference between pre-training and fine-tuning losses is foundational for interviews. Pre-training uses self-supervised objectives — the label comes from the data itself. For autoregressive models (GPT family, Llama, Mistral), this is next-token prediction:

Lpretrain(θ) = − Σt=1T log P(xt | x<t; θ)

Every token is predicted given all preceding tokens. The loss sums over all positions — the model learns from every single token in the corpus. This is extraordinarily data-efficient because a sequence of length T provides T training signals.

For encoder models (BERT family), pre-training uses masked language modeling (MLM):

LMLM(θ) = − Σt ∈ masked log P(xt | x\masked; θ)

Only ~15% of tokens are masked and predicted, using bidirectional context. This produces better representations for classification but can't generate text autoregressively.

Supervised fine-tuning (SFT) for instruction-following shifts to a conditional generation loss. Given an instruction-response pair (x, y), we backpropagate only through response tokens:

LSFT(θ) = − Σt=1|y| log P(yt | x, y<t; θ)
🧮 Derive It Yourself — Next-Token Prediction Loss (All Variants)

Every loss function in this section is a variation of one idea. Master the pattern once and you can write any of them:

1

Start from probability: An autoregressive model defines P(sequence) = Πt P(xt | x<t). This is the chain rule of probability — nothing model-specific.

2

Take the log: log P(sequence) = Σt log P(xt | x<t). Products become sums. This is the log-likelihood.

3

Negate it (to get a loss): L = −Σt log P(xt | x<t). We minimize loss = maximize likelihood. This is the pre-training loss — summed over ALL tokens.

4

Choose which tokens to sum over — that's the only difference:

• Pre-training: Sum over ALL positions t = 1...T → learns to predict every token
• MLM (BERT): Sum only over masked positions → bidirectional but can't generate
• SFT: Sum only over response tokens, condition on prompt x → learns to answer, not to parrot
• Multi-turn SFT: Sum over assistant tokens across all turns → learns multi-turn conversation

Whiteboard shortcut: Write "L = −Σ log P(token | context)" and then just define which tokens are in the sum and what counts as "context." Every language model loss is this formula with different bookkeeping.
🚨 Critical Nuance: Prompt Masking

Why mask prompt tokens? If we include prompt tokens in the loss, the model learns to predict the next token in the prompt — essentially learning to generate instructions rather than respond to them. This leads to models that echo or rephrase the question instead of answering it. In practice, the loss is masked to zero for all prompt positions and computed only over the response span. This is implemented in frameworks via labels = [-100] * len(prompt_ids) + response_ids where -100 is the ignore index for cross-entropy loss.

Common mistake in production: Some training scripts accidentally include the system prompt or few-shot examples in the loss computation. This wastes capacity on learning to generate boilerplate text and can degrade response quality by up to 5–10% on benchmarks.

1.3 The Fine-Tuning Spectrum

Modern fine-tuning is not binary — it exists on a continuum from minimal parameter modification (prompt tuning) to full weight updates. Understanding where each method sits and why you'd choose it is a core interview topic.

🔬 Why does this spectrum exist?

The theoretical justification comes from the intrinsic dimensionality hypothesis (Aghajanyan et al., 2021). Pre-trained models have very low intrinsic dimensionality for downstream tasks — meaning the effective subspace you need to modify is tiny relative to the full parameter space. A 355M-parameter model was shown to need only ~200 dimensions for 90% of full fine-tuning performance on many tasks. As models get larger, this intrinsic dimensionality grows sublinearly, which is why PEFT methods become even more effective at scale.

Intuitively: pre-training has already positioned the model near many good solutions for downstream tasks. Fine-tuning only needs to make a small correction in parameter space to reach the specific solution for your task. The larger the model, the closer it already is to those solutions, so the correction is proportionally smaller.

1.4 When to Fine-Tune vs. Alternatives

✅ Fine-Tune When
  • Consistent behavior needed: You need the model to always follow a specific output format (JSON schema, specific markdown structure, etc.) — prompting is brittle for this
  • Domain-specific knowledge: The base model hasn't seen your domain data (proprietary docs, niche scientific fields, specific coding frameworks)
  • Latency constraints: Internalizing knowledge avoids retrieval round-trips; shorter prompts reduce time-to-first-token
  • Alignment/safety: You need consistent policy adherence that can't be jailbroken through prompt injection
  • Cost at scale: Fine-tuned models with shorter prompts cost less per inference call than few-shot prompting
⏸️ Consider Alternatives When
  • Few-shot works well enough: If 3–5 examples in the prompt achieve >90% of target quality, don't fine-tune
  • Factual grounding needed: RAG is better for current/evolving knowledge — fine-tuned facts become stale and can hallucinate confidently
  • Insufficient data: <100 examples for complex tasks; <500 for reasoning tasks. The model will memorize rather than generalize
  • Rapid iteration: If you're still figuring out what "good" looks like, prompting lets you iterate in minutes vs. hours for fine-tuning
  • Multi-task flexibility: If you need one endpoint for many diverse tasks, fine-tuning can narrow capabilities
💡 The Hybrid Pattern: Fine-Tuning + RAG

In production, the most robust systems combine fine-tuning and RAG. Fine-tune for: output format compliance, reasoning style, safety guardrails, and domain-specific language/jargon. Use RAG for: factual answers, time-sensitive information, and cite-able sources. This avoids the biggest risk of fine-tuning on factual knowledge — the model will confidently assert outdated or wrong facts with no mechanism to self-correct.

Example architecture: Fine-tune a Llama-3 model on 10K examples of your company's documentation Q&A format (teaching it how to cite sources, format answers, handle multi-part questions). At inference, use RAG to retrieve relevant docs and inject them into the context. The fine-tuned model produces well-formatted, consistent answers; RAG ensures factual accuracy and currency.

02 Full Fine-Tuning: Theory & Practice

2.1 Optimization Formulation

Full fine-tuning updates every single parameter of the model. For a transformer with L layers, each containing multi-head attention (MHA) and feed-forward network (FFN) sublayers, the total parameter count scales as:

||θ|| ≈ L × (4d² + 2d·dff + dff·d) + V × d (embedding layer)

where d is the hidden dimension, dff is the FFN intermediate dimension (4d for standard, ~2.67d for SwiGLU-style which uses 3 weight matrices instead of 2), and V is vocabulary size. The 4d² in attention comes from WQ, WK, WV, WO each being d×d (assuming n_heads × head_dim = d). For a 7B model with d=4096, dff=11008 (SwiGLU), L=32, this gives ~6.7B dense parameters plus ~0.5B in embeddings.

2.2 AdamW: The Standard Optimizer

Nearly all LLM fine-tuning uses AdamW (Adam with Decoupled Weight Decay). Understanding its internals is essential:

mt = β₁ · mt-1 + (1 − β₁) · gt ← first moment (momentum/mean)
vt = β₂ · vt-1 + (1 − β₂) · gt² ← second moment (variance)
m̂t = mt / (1 − β₁t) ← bias correction
v̂t = vt / (1 − β₂t) ← bias correction
θt = θt-1 − η · (m̂t / (√v̂t + ε) + λ · θt-1)
🔬 Why AdamW over Adam + L2?

In vanilla Adam with L2, the regularization term λ·θ is added to the gradient before the adaptive step. This means the regularization effect is scaled by 1/√v̂ — parameters with large gradient variance get less regularization, and parameters with small variance get more. This is undesirable and inconsistent.

AdamW decouples weight decay from the gradient-based update. The decay λ·θ is applied directly to the weights after the adaptive gradient step. This ensures uniform regularization regardless of gradient history. The result is more consistent training dynamics and better generalization, particularly important for fine-tuning where you want controlled, uniform regularization to prevent overfitting.

Typical values for fine-tuning: β₁ = 0.9, β₂ = 0.999 (some use 0.95 for instability), ε = 1e-8, λ = 0.01–0.1. The bias correction terms matter for early training steps — without them, m̂ and v̂ are biased toward zero since m₀ = v₀ = 0.

🧮 Derive It Yourself — AdamW from SGD

Build the entire AdamW update rule by adding one ingredient at a time:

1

Start with SGD: θt = θt-1 − η · gt. Simplest optimizer: move in the direction of the negative gradient. Problem: noisy, oscillates, same learning rate for all parameters.

2

Add momentum (→ SGDM): Average past gradients with an exponential moving average: mt = β₁ · mt-1 + (1−β₁) · gt. Now θt = θt-1 − η · mt. This smooths out noise and accelerates through flat regions. β₁ = 0.9 means "remember 90% of previous direction."

3

Add per-parameter scaling (→ Adam): Track the second moment: vt = β₂ · vt-1 + (1−β₂) · gt². Divide the update by √vt: parameters with large gradients get smaller updates, parameters with small gradients get larger updates. This is adaptive learning rates. Now: θt = θt-1 − η · mt / (√vt + ε).

4

Fix the initialization bias (→ Adam with bias correction): Since m₀ = v₀ = 0, early estimates are biased toward zero. Correct by dividing: m̂t = mt/(1−β₁t), v̂t = vt/(1−β₂t). The correction is large at t=1 and vanishes as t→∞. Why (1−βt)? Unroll the EMA: mt = (1−β₁)·Σi=1..t β₁t-i·gi. The sum of the weights is (1−β₁)·(1−β₁t)/(1−β₁) = 1−β₁t, not 1. Dividing normalizes it.

5

Decouple weight decay (→ AdamW): Instead of adding λ·θ to the gradient before the adaptive step (which would scale the regularization by 1/√v̂ — inconsistent!), apply it after: θt = θt-1 − η · (m̂t/(√v̂t+ε) + λ · θt-1). Every parameter decays at the same rate λ, regardless of gradient history.

Whiteboard strategy: Write SGD first. Add momentum. Add adaptive second moment. Add bias correction. Add decoupled decay. Each step addresses one problem with the previous version. If you can explain why each addition helps, you demonstrate deep understanding.

2.3 Mixed-Precision Training: The Details

Mixed-precision maintains a master copy in fp32 while doing compute in lower precision. The full protocol:

  1. Store master weights in fp32. This is the source of truth. All optimizer state updates happen here. Keeping fp32 prevents the gradual accumulation of rounding errors that would degrade training over thousands of steps.
  2. Cast to bf16/fp16 for forward pass. θ̂ = cast(θ). This halves memory for activations and speeds up matmuls (Tensor Cores operate on fp16/bf16).
  3. Compute loss in fp32. The loss computation (log-softmax over vocabulary) is numerically sensitive. Computing it in fp16 causes significant errors, especially for low-probability tokens.
  4. Backward pass in bf16/fp16. Gradients are computed in reduced precision. With bf16, this is generally safe. With fp16, you need loss scaling: multiply the loss by a large factor (e.g., 1024) before backward, then divide gradients by the same factor. This prevents small gradients from underflowing to zero.
  5. Update master weights in fp32. Convert gradients to fp32, apply the AdamW update to fp32 master weights.
🔬 bf16 vs. fp16: A Detailed Comparison

fp16 (IEEE half-precision): 1 sign bit + 5 exponent bits + 10 mantissa bits. Range: ~6×10⁻⁸ to ~65,504. The narrow range means gradients smaller than 6×10⁻⁸ become zero (underflow), and values above 65,504 become infinity (overflow). This is why loss scaling is mandatory — it shifts small gradients into the representable range.

bf16 (brain float 16): 1 sign bit + 8 exponent bits + 7 mantissa bits. Range: ~1.2×10⁻³⁸ to ~3.4×10³⁸ — the same as fp32. The wider range eliminates underflow/overflow issues, so no loss scaling needed. The trade-off is lower precision (7 mantissa bits vs. 10), meaning each number has less accuracy. In practice, this lower precision doesn't matter for training because gradient noise dominates rounding errors.

For LLM fine-tuning: bf16 is strongly preferred. It's simpler (no loss scaler to manage), more stable (no overflow/underflow edge cases), and produces equivalent results. The only reason to use fp16 is if your hardware doesn't support bf16 (pre-Ampere NVIDIA GPUs, i.e., V100 and older).

Memory savings with mixed precision: For a 7B model, parameters in bf16 = 14GB instead of 28GB. Activations are also halved. However, optimizer states remain fp32, so the total savings are typically ~30–40% vs. pure fp32 training.

2.4 Gradient Checkpointing (Activation Recomputation)

During backpropagation, you need the forward-pass activations at each layer to compute gradients. For a transformer, the activation memory scales as:

Activation memory ≈ L × B × S × d × k (where k ≈ 10–12 for a standard transformer layer)

The factor k accounts for multiple intermediate tensors: attention queries/keys/values, attention weights, attention output, FFN intermediate, residual connections, and layer norms. For a 7B model with batch size 4 and sequence length 2048, this can be 40–60 GB of activation memory — often more than the model parameters themselves.

Gradient checkpointing (also called activation recomputation or rematerialization) addresses this by only storing activations at selected "checkpoint" layers and recomputing the others during the backward pass. With checkpoints every k layers:

Memory: O(L/k × B × S × d) (reduced by factor k)
Compute: O(L × k) (increased by recomputing k layers per checkpoint)
Optimal k = √L (~5–6 for L=32)
🧮 Derive It Yourself — Optimal Checkpoint Spacing

This is a classic memory-compute trade-off optimization:

1

Setup: L layers total. Place checkpoints every k layers → L/k checkpoints stored. Each stores activations of size C per layer. Memory = (L/k) · C.

2

Compute cost: During backward pass, for each checkpoint segment, you re-run the forward pass through k layers to recompute activations. Total recomputation ≈ L layers (one re-forward per segment, each segment has k layers, there are L/k segments → L re-computations total). But the recomputation overhead on top of the normal backward pass is proportional to L — i.e., ~one extra forward pass.

3

Wait — what about non-uniform checkpointing? If we don't checkpoint every k layers but choose optimally with a budget of m checkpoints, the memory is proportional to m + L/m (m checkpoints + max segment length). Minimize total cost = αm + βL/m. Take derivative, set to 0: α − βL/m² = 0 → m* = √(βL/α). If costs are equal (α=β): m* = √L, giving optimal spacing k = √L.

Interview-ready version: "Gradient checkpointing is a memory-compute trade-off. Store activations every k layers: memory scales as L/k, extra compute scales as k (recomputing k layers per segment). Total cost ~ L/k + k, minimized at k = √L by AM-GM inequality. For a 32-layer model, checkpoint every ~6 layers."
⚡ Practical Impact

Gradient checkpointing typically reduces activation memory by 60–70% at the cost of 30–40% more compute time. This is almost always worthwhile for fine-tuning because: (1) it lets you use larger batch sizes, which improves gradient quality, and (2) it can make the difference between fitting training on your available GPUs or not. In PyTorch, enable it with model.gradient_checkpointing_enable(). In HuggingFace Trainer, set gradient_checkpointing=True in TrainingArguments.

2.5 Flash Attention

Dao et al., 2022 — FlashAttention Dao, 2023 — FlashAttention-2

Standard attention materializes the full S×S attention matrix, using O(S²) memory. Flash Attention computes attention in a tiled, fused manner that never materializes the full matrix, reducing memory to O(S) and improving speed by 2–4x through better GPU memory hierarchy utilization (keeping data in SRAM rather than repeatedly reading from HBM).

For fine-tuning, Flash Attention is practically mandatory for sequences longer than 2048 tokens. Combined with gradient checkpointing, it can reduce total memory usage by 5–10x compared to naive implementations. FlashAttention-2 further optimizes parallelism across sequence length and heads, achieving near-theoretical peak throughput on modern GPUs.

2.6 Practical Hyperparameters — The Full Story

  • Learning rate (1e-5 to 5e-5): This is 10–100× smaller than pre-training LR. The intuition: pre-training starts from random initialization and needs large updates to move through parameter space quickly; fine-tuning starts from a good initialization and needs small, careful adjustments. Too high → catastrophic forgetting (obliterates pre-trained knowledge); too low → undertrained, wasting compute. Interview tip: If asked "what's the first thing you'd try if fine-tuning isn't working?", the answer is usually "adjust the learning rate."
  • Learning rate warmup (3–10% of total steps): The first few gradient steps are noisy because they're computed from a small sample. Without warmup, these noisy gradients are applied at full learning rate, potentially pushing parameters far from the good initialization. Linear warmup from 0 to peak LR over the first 3–10% of steps prevents these destructive early updates. This is especially important for fine-tuning because the pre-trained initialization is valuable — you don't want to destroy it with the first few updates.
  • LR scheduler — cosine decay: After warmup, LR decays following LR(t) = LRmin + 0.5 × (LRmax - LRmin) × (1 + cos(πt/T)). Decays to ~10% of peak. Why cosine? Early training benefits from higher LR (exploring loss landscape), while late training benefits from lower LR (converging precisely). Cosine decay also provides a gradual transition, avoiding the sudden drops of step-decay schedulers. In practice, cosine decay gives ~0.5–1% better final performance than linear decay.
  • Batch size (effective 32–256): Larger batches produce lower-variance gradient estimates, leading to more stable training. But too large a batch can hurt generalization (the "generalization gap"). For fine-tuning, batch sizes of 32–128 work well. If memory-limited, use gradient accumulation: accumulate gradients over N micro-batches before an optimizer step. Effective batch size = micro_batch_size × num_GPUs × gradient_accumulation_steps.
  • Epochs (1–3 for large datasets, 3–10 for small): More epochs means seeing each example more times. With large datasets (>100K examples), 1–2 epochs is often optimal — the model learns the distribution quickly and additional passes cause overfitting. With small datasets (<10K), 3–5 epochs may be needed, but monitor eval loss closely for the inflection point where overfitting begins.
  • Max sequence length: Should match your inference use case. Longer sequences consume more memory (quadratically for attention, linearly for other components with Flash Attention). For fine-tuning, it's common to pack multiple short examples into one sequence (with proper attention masking) to maximize GPU utilization.
  • Gradient clipping (max_grad_norm = 1.0): Clips the global gradient norm to prevent single large-gradient steps from destabilizing training. This is a safety net — in well-configured training, gradients rarely exceed the threshold, but when they do (due to outlier examples or numerical issues), clipping prevents catastrophic parameter updates.

03 Parameter-Efficient Fine-Tuning (PEFT)

3.1 Why PEFT? The Deep Motivation

As models scale to tens and hundreds of billions of parameters, full fine-tuning becomes impractical for three reasons: (1) Memory: A 70B model needs ~1.1TB for full fine-tuning (parameters + optimizer states + gradients in fp32). (2) Storage: Each fine-tuned version is a full copy of the model. Serving 100 task-specific variants of a 70B model means storing 100 × 140GB = 14TB. (3) Catastrophic forgetting: Modifying all parameters makes it easy to overwrite the valuable pre-trained knowledge.

PEFT methods address all three by modifying a tiny subset of parameters. The formal framework:

Effective parameters: θeff = θ₀ + Δθ(φ)
where θ₀ ∈ ℝd is frozen, φ ∈ ℝm is trainable, m ≪ d
Δθ: ℝm → ℝd is the reparameterization function

The core hypothesis: for any given downstream task, the optimal weight change Δθ* lies in a low-dimensional subspace of ℝd. This is supported by the intrinsic dimensionality findings — and it turns out to be even more true at larger scales. A 175B model may need a lower percentage of parameters modified than a 7B model for the same task.

🧮 Derive It Yourself — The PEFT Reparameterization Framework

Every PEFT method is an instance of one general framework. Derive it once, then plug in specifics:

1

Start from the problem: We want to update weights but not modify W₀ directly (too expensive, too risky). So express the effective weight as: Weff = W₀ + ΔW. Here ΔW is the change we want to learn.

2

ΔW is huge (same size as W₀). Compress it: Instead of learning ΔW ∈ ℝd×k directly (~millions of params), parameterize it through a small set of trainable parameters φ ∈ ℝm (m ≪ d×k) via some function: ΔW = f(φ). Now you only optimize φ.

3

Different PEFT methods = different choices of f(φ):

• LoRA: f(φ) = BA where φ = {B, A}, rank-r factorization → m = r(d+k)
• Adapters: f(φ) = Wup · σ(Wdown · h) — but applied as a residual addition to activations rather than to weights
• Prefix Tuning: f(φ) = learnable key/value vectors prepended to attention — modifies the input to attention rather than the weights
• VeRA: f(φ) = diag(db) · Brandom · diag(da) · Arandom — φ is just two scaling vectors
• BitFit: f(φ) = bias terms only — φ is the set of all biases

4

The forward pass is always: h = (W₀ + ΔW)x = W₀x + f(φ)·x. The gradient flows only through φ (W₀ is frozen). This is why PEFT is memory-efficient: optimizer states are proportional to |φ|, not |W₀|.

Interview power move: When asked about any PEFT method, first write the general framework Weff = W₀ + f(φ) and then say "the specific method defines f(φ) as..." This shows you see the unity across methods and can reason about new ones from first principles.

3.2 Additive Methods — Detailed Theory

Adapters (Houlsby et al., 2019)

Houlsby et al., 2019 — Parameter-Efficient Transfer Learning

Adapters insert small bottleneck MLP modules after each transformer sublayer (attention and FFN). The architecture is a down-projection, nonlinearity, up-projection, plus a residual connection:

h ← h + Wup · σ(Wdown · h)
where Wdown ∈ ℝd×r, Wup ∈ ℝr×d, r ≪ d

Why the bottleneck? The down-projection compresses the d-dimensional hidden state to r dimensions, forcing the adapter to learn a compact, low-dimensional representation of the task-specific transformation. The nonlinearity σ (typically ReLU or GELU) allows nonlinear transformations that a simple linear layer couldn't capture. The up-projection maps back to d dimensions. The residual connection ensures that if the adapter learns nothing useful (all-zero output), the model behaves identically to the pre-trained base.

Parameter count: Each adapter adds 2 × r × d + r (bias) parameters. With r = 64 and d = 4096, that's ~524K per insertion point. A 32-layer transformer with adapters after both attention and FFN sublayers adds 32 × 2 × 524K ≈ 33.5M parameters — about 0.5% of a 7B model.

Downside: Adapters add sequential computation that cannot be merged into existing weights, introducing inference latency (typically 5–15%). This is their main disadvantage vs. LoRA, which can be merged for zero-overhead inference.

Prefix Tuning (Li & Liang, 2021)

Li & Liang, 2021 — Prefix-Tuning

Prepends learnable continuous vectors ("virtual tokens") to the key and value matrices at every attention layer. For prefix length p and L layers:

K' = [PK(l) ; K(l)], V' = [PV(l) ; V(l)] ∀ layer l
Attn(Q, K', V') = softmax(QK'T / √dk) · V'
Trainable params = L × 2 × p × d

Key insight: By only modifying K and V (not Q), prefix tuning steers what the model attends to without changing what generates the queries. The prefix acts as a "virtual context" that biases attention patterns toward task-relevant representations.

Reparameterization trick: Directly optimizing the prefix vectors is unstable (high-dimensional, unconstrained optimization). Instead, Li & Liang parameterize the prefixes through a small MLP: P = MLP(Pbase) during training. After training, the MLP is discarded and only the final prefix vectors are kept. This is analogous to how VAEs use an encoder to produce latent representations — the reparameterization provides a smoother optimization landscape.

Downside: The prefix tokens consume sequence length budget. If your model supports 4096 tokens and your prefix is 128 tokens, you effectively have 3968 tokens for actual input. This matters for long-context applications.

Prompt Tuning (Lester et al., 2021)

Lester et al., 2021 — The Power of Scale for Prompt Tuning

The simplest PEFT method: prepend learnable soft prompt embeddings only at the input layer. No modification to intermediate layers.

Trainable params = p × d (just p vectors of dimension d)

For p=20 and d=4096, this is only 80K parameters — less than 0.001% of a 7B model. The remarkable finding from Lester et al. is that prompt tuning closes the gap with full fine-tuning as model size increases. At 11B parameters (T5-XXL), prompt tuning achieves nearly identical performance to full fine-tuning on SuperGLUE. At smaller scales (under 1B), it's significantly worse.

Why does scale help? Larger models have more expressive attention mechanisms that can extract richer information from the soft prompt. The soft prompt acts as a compressed task descriptor, and a more capable model can "decode" it more effectively.

Selective Methods

  • BitFit (Ben-Zaken et al., 2022): Trains only bias terms (~0.1% of parameters). Surprisingly competitive on NLU tasks (GLUE benchmarks). The intuition: biases control the "activation threshold" of neurons, and shifting these thresholds can significantly change model behavior without altering the learned feature representations in the weight matrices.
  • Layer-selective fine-tuning: Freeze lower layers, train upper layers. Justified by the observation that lower layers learn universal features (tokenization, basic syntax, part-of-speech) while upper layers learn task-specific features (semantics, reasoning patterns). For a 32-layer model, fine-tuning only the top 8 layers modifies ~25% of parameters while capturing most of the task-specific adaptation.

04 LoRA & Its Variants

4.1 LoRA: Low-Rank Adaptation — Full Theory

Hu et al., 2022 — LoRA microsoft/LoRA huggingface/peft

LoRA is the most widely used PEFT method and understanding it deeply is essential for any Gen AI researcher interview.

Core insight: The weight updates ΔW during fine-tuning have low intrinsic rank. Instead of learning a dense update to W ∈ ℝd×k, decompose it into two low-rank matrices:

W' = W₀ + ΔW = W₀ + B · A
where B ∈ ℝd×r, A ∈ ℝr×k, r ≪ min(d, k)

Forward pass: h = W₀x + (α/r) · B(Ax)

Why this works mathematically: Any matrix ΔW ∈ ℝd×k can be decomposed via SVD as ΔW = UΣVT where U ∈ ℝd×d, Σ is diagonal, V ∈ ℝk×k. If the singular values decay rapidly (i.e., σ₁ ≫ σ₂ ≫ ... ≫ σr ≫ σr+1 ≈ 0), then ΔW is well-approximated by its rank-r truncation: ΔW ≈ UrΣrVrT = BA where B = UrΣr1/2 and A = Σr1/2VrT. Hu et al. verified empirically that learned ΔW matrices have rapidly decaying singular values — the top 1–4 singular values capture the majority of the "useful" update, with the rest being noise.

🧮 Derive It Yourself — LoRA from First Principles

Here's how to derive LoRA from scratch on a whiteboard, starting from the SVD insight:

1

Observation: When we full-fine-tune and compute ΔW = Wfinetuned − Wpretrained, the SVD shows that ΔW has low rank. Most singular values are near zero. Formalize: rank(ΔW) ≈ r where r ≪ min(d,k).

2

If ΔW is low-rank, factor it: Any rank-r matrix can be written as a product of two thin matrices: ΔW = B · A where B ∈ ℝd×r and A ∈ ℝr×k. Parameter savings: d×k → d×r + r×k = r(d+k). For d=k=4096, r=16: from 16.7M to 131K — a 128× reduction.

3

Design the forward pass: h = W₀x + BAx. Since W₀ is frozen, only B and A receive gradients. Memory: optimizer states scale with r(d+k) instead of d×k.

4

Handle the scaling problem: The norm of BAx depends on r (larger r → larger output norm on average). We want changing r to not change the scale of the output. So add a scaling factor: h = W₀x + (α/r) · BAx. Now the effective output magnitude is controlled by α (a hyperparameter), independent of r.

5

Choose initialization: We need ΔW = BA = 0 at start (to preserve pre-trained behavior). Set B = 0. Keep A random (Kaiming) so that Ax gives diverse projections — this provides a non-zero gradient signal for B immediately: ∂L/∂B = (∂L/∂(BAx)) · (Ax)T ≠ 0 since Ax ≠ 0.

6

At inference, merge: W' = W₀ + (α/r)BA. This is a single d×k matrix. No architectural change, no latency overhead. This is the key advantage over adapters (which add sequential computation) and prefix tuning (which consumes context length).

The SVD connection is key: If an interviewer asks "why low-rank?", draw the singular value spectrum: σ₁ ≫ σ₂ > σ₃ > ... > σr ≫ σr+1 ≈ 0. The rapid decay means the true ΔW lives in a rank-r subspace. LoRA simply searches within that subspace directly, rather than searching the full d×k space and hoping to land in the subspace.
⚡ Initialization — Why It Matters Deeply

A is initialized with Kaiming uniform (random Gaussian with appropriate variance) and B is initialized to all zeros. This is a deliberate and critical design choice:

  • ΔW = BA = 0 at initialization — the model starts with exact pre-trained behavior. This means LoRA is a strictly conservative modification: if training doesn't find a useful update, the model falls back to the base model exactly.
  • Why not initialize both randomly? Random BA would produce a random perturbation to the pre-trained weights, potentially destroying the good initialization. Zero-init of one matrix guarantees no initial damage.
  • Why zero B and not A? The gradient of the loss with respect to B is ∂L/∂B = (∂L/∂(BAx)) · (Ax)T. Since A is random, Ax produces diverse projections of the input, providing a rich gradient signal for B from the first step. If A were zero instead, BAx = 0 regardless of B, and the gradient for A would be BT · (∂L/∂h) = 0 — training would never start.

The Scaling Factor α/r

The scaling factor α/r controls the magnitude of the adaptation. Its purpose and tuning is a common source of confusion:

h = W₀x + (α/r) · BAx
  • Why divide by r? When you increase rank, the norm of BAx tends to increase proportionally to √r (by random matrix theory, since A is randomly initialized). Dividing by r compensates, ensuring that the scale of the adapter output doesn't change dramatically when you change r. This makes hyperparameter transfer across ranks easier.
  • α is a separate tuning knob that controls the overall scale. Common heuristics: α = r (effective scaling of 1), α = 2r (effective scaling of 2, more aggressive adaptation), or α = 16 as a fixed value regardless of r.
  • The effective learning rate for LoRA is approximately η × α/r. This means if you double r, you should also double α to keep the effective learning rate constant. If you change α, you implicitly change the learning rate for the adapter parameters. This interaction is the source of many failed experiments.

Which Modules to Target?

The original paper only applied LoRA to WQ and WV in attention layers. Subsequent research has conclusively shown:

💡 Module Selection Best Practice

Apply LoRA to all linear layers (Q, K, V, O in attention + up, gate, down in MLP) at a lower rank per module. For a fixed total parameter budget, this consistently outperforms applying LoRA to fewer modules at a higher rank. The intuition: each layer needs only a small correction, and spreading the capacity across all layers captures more diverse adaptations than concentrating it in a few layers.

Practical config for a 7B model: target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "up_proj", "gate_proj", "down_proj"], r=32, alpha=64. This gives ~82M trainable parameters (~1.2% of model) and matches or exceeds full fine-tuning on most benchmarks.

Merging at Inference

After training, the adapter can be folded back into the base weights: W' = W₀ + (α/r)BA. This is a single matrix addition that produces a new weight matrix. At inference, the merged model has identical architecture and latency to the original — no extra parameters, no extra computation. This is LoRA's killer feature vs. adapters or prefix tuning.

Multi-adapter serving: You can maintain one base model in GPU memory and dynamically load different LoRA adapters (which are tiny — ~100MB for a 7B model at r=32). This enables multi-tenant serving where different users get different fine-tuned behaviors from a single base model.

🔬 How to Empirically Determine the Right Rank

Method 1 — Singular value analysis: After training at a high rank (e.g., r=256), compute the SVD of each BA product. Plot the singular values. If they decay to near-zero by rank 16, you're over-parameterized — r=16 would suffice. If they're still significant at rank 256, you may need higher rank.

Method 2 — Rank sweep: Train at r ∈ {4, 8, 16, 32, 64, 128} and plot validation loss. There's typically a clear elbow where increasing rank stops helping. This is the most reliable method but requires multiple training runs.

Method 3 — Dynamic rank (AdaLoRA): Zhang et al., 2023 — AdaLoRA automatically allocates rank across layers based on importance scores. Layers that need more adaptation get higher rank; layers that are already well-suited get lower rank. This can squeeze more performance from a fixed parameter budget.

Important nuance: Increasing rank beyond what the task requires doesn't just plateau — it can actively hurt performance through overfitting. The extra capacity memorizes training noise rather than learning generalizable patterns. This is counterintuitive to people who expect "more parameters = better" but is consistently observed in practice.

4.3 LoRA Variants — Comprehensive Guide

QLoRA (Dettmers et al., 2023)

Dettmers et al., 2023 — QLoRA artidoro/qlora

QLoRA is arguably the most impactful practical contribution to fine-tuning accessibility. It combines LoRA with aggressive quantization of the base model, enabling fine-tuning of models that would otherwise require enterprise-grade hardware.

Three key innovations:

  1. 4-bit NormalFloat (NF4): A non-uniform quantization data type optimally designed for normally-distributed weights. Neural network weights empirically follow a near-normal distribution. Standard uniform INT4 spaces 16 quantization bins equally across the value range, but since weights concentrate around zero, most bins cover sparse regions while the dense center is under-represented. NF4 solves this by placing bin edges at the quantiles of N(0, σ²) — each of the 16 bins has equal probability mass under the normal distribution. This is provably information-theoretically optimal (minimizes expected quantization error) for normally-distributed data. The improvement: ~30–50% less quantization error than uniform INT4.
  2. Double quantization: NF4 requires per-block scaling factors (one fp32 scale per 64 parameters = 0.5 bits/param overhead). Double quantization quantizes these scaling factors themselves from fp32 to fp8, reducing the overhead to 0.127 bits/param. This seems small, but for a 70B model, it saves ~3.3GB — significant when you're memory-constrained.
  3. Paged optimizers: Uses NVIDIA unified memory (managed CUDA memory) to automatically page optimizer states between GPU and CPU memory. When a gradient spike temporarily requires more memory than available GPU VRAM, states are paged to CPU rather than crashing with OOM. This is transparent to the training loop.
⚡ How Gradients Flow Through Quantized Weights

During the forward pass, quantized base weights are dequantized on-the-fly: Wdequant = scale × (W4bit - zero_point). The LoRA adapters operate in bf16: h = Wdequantx + (α/r) · BAx. During the backward pass, gradients flow through the dequantization operation using the straight-through estimator — the gradient of the dequantization is treated as identity. Crucially, only the LoRA parameters (A, B) receive optimizer updates; the quantized base weights are never modified.

Result: A 65B model can be fine-tuned on a single 48GB A6000 GPU. A 7B model fits on a consumer RTX 3090 (24GB). A 13B model fits on an RTX 4090 (24GB). This democratized fine-tuning enormously.

DoRA — Weight-Decomposed Low-Rank Adaptation

Liu et al., 2024 — DoRA
W' = m · (W₀ + BA) / ||W₀ + BA||c

DoRA decomposes weight matrices into magnitude and direction components, inspired by weight normalization (Salimans & Kingma, 2016). The insight: LoRA's update BA simultaneously changes both the magnitude and direction of weight vectors. These are conceptually different operations — scaling (how strongly the layer responds) vs. rotation (what features the layer computes) — and conflating them limits LoRA's expressiveness.

In DoRA, m is a learnable magnitude vector (one scalar per output neuron), and the direction is given by the normalized (W₀ + BA). The magnitude and direction are optimized independently, allowing more precise adaptation. Empirically, DoRA improves over LoRA by 1–3% across tasks with negligible parameter increase (just d additional scalars for the magnitude vector).

rsLoRA — Rank-Stabilized LoRA

Kalajdzievski, 2024

Standard LoRA's scaling α/r causes the effective learning rate to depend on rank. The analysis: when A is initialized with variance 1/r (Kaiming), the expected norm of BAx scales as √r (by random matrix theory). With scaling α/r, the adapter output scales as α/√r — which decreases with rank. This means higher rank actually starts with weaker adaptation signal.

rsLoRA corrects this by using α/√r as the scaling factor. Since ||BAx|| ∝ √r, the product (α/√r) × √r = α remains constant regardless of rank. This stabilization makes hyperparameter transfer across ranks much more reliable — you can change rank without needing to retune learning rate or α.

🧮 Derive It Yourself — Why α/r Is Wrong and α/√r Is Right

This requires a bit of random matrix theory, but the argument is clean:

1

At initialization: B = 0 and A has entries ~ N(0, 1/r) (Kaiming initialization for a d→r mapping). After a few gradient steps, B also has non-trivial entries. Consider the scale of BAx.

2

Compute ||Ax||: A ∈ ℝr×k with entries ~ N(0, 1/r). For input x ∈ ℝk, each component of Ax is a sum of k terms, each ~ N(0, ||x||²/r). So each component of Ax has variance ||x||²/r, and ||Ax||² ≈ r · ||x||²/r = ||x||². So ||Ax|| ≈ ||x|| — independent of r (by Kaiming design).

3

Compute ||BAx||: After training, B entries have some scale σB. ||BAx||² ≈ d · σB² · ||Ax||² ≈ d · σB² · ||x||². But wait — this ignores the accumulation across r dimensions. The matrix product BAx sums r terms (one per LoRA dimension). By CLT, the norm scales as √r. So: ||BAx|| ∝ √r · ||x||.

4

With standard LoRA (α/r scaling): The adapter output is (α/r) · BAx. Its norm ∝ (α/r) · √r = α/√r. This shrinks with r! Doubling r halves the effective adapter contribution — counterintuitive and harmful.

5

With rsLoRA (α/√r scaling): The adapter output is (α/√r) · BAx. Its norm ∝ (α/√r) · √r = α. Constant regardless of r. Now changing rank doesn't change the initialization scale, so hyperparameters transfer cleanly.

The quick version for interviews: "BAx norm scales as √r due to the sum over r dimensions. Standard LoRA divides by r, making the effective signal shrink as 1/√r with rank. rsLoRA divides by √r instead, keeping the signal constant. This is why rsLoRA enables clean hyperparameter transfer across ranks."

LoRA+ (Hayou et al., 2024)

Hayou et al., 2024

Observes that the optimal learning rates for A and B matrices differ significantly. The theoretical analysis uses random matrix theory: for model width d, the gradient norm for A scales as O(1) while for B it scales as O(1/d). With a fixed learning rate, B receives effectively too-small updates relative to A.

Solution: Use ηB = λ × ηA with λ ≈ 16. This compensates for the scale difference and improves convergence speed by 1.5–2× with no additional compute cost. Simple to implement — just assign different parameter groups in the optimizer.

VeRA — Vector-based Random Matrix Adaptation

Kopiczko et al., 2024
ΔW = Λb · Bshared · Λa · Ashared
where Λb = diag(db), Λa = diag(da), and Ashared, Bshared are frozen random matrices shared across ALL layers

VeRA pushes parameter efficiency to the extreme. Instead of learning separate A and B for each layer (which is the LoRA approach), VeRA generates random Ashared and Bshared matrices once (shared globally across all layers) and learns only small per-layer scaling vectors da and db. The trainable parameters per layer are just 2 × r scalars instead of (d + k) × r for LoRA. For r = 256, d = 4096, k = 4096, LoRA has 2M params/layer while VeRA has just 512 — a ~4000× reduction. Total trainable parameters can be as low as 0.01% of the model.

Why does this work? Random projection theory (Johnson-Lindenstrauss lemma) shows that random matrices preserve geometric structure. The shared random matrices provide a rich enough basis that per-layer scaling can select the appropriate subspace. It's lossy compared to LoRA (each layer has less freedom) but the savings are enormous, making it ideal for extremely memory-constrained settings.

GaLore — Gradient Low-Rank Projection

Zhao et al., 2024 jiaweizzhao/GaLore
Every T steps: compute SVD of gradient G, extract P = top-r left singular vectors
For each step: Gproj = PT · G (project gradient to rank-r subspace)
Optimizer states maintained in rank-r space: m, v ∈ ℝr×k instead of ℝd×k

GaLore takes a fundamentally different approach from LoRA: instead of constraining the weights to be low-rank, it constrains the gradients. The full weight matrix is updated, but through a low-rank gradient subspace. The subspace (defined by P) is periodically refreshed every T steps (typically T = 200) to track the changing optimization landscape.

Advantages over LoRA: (1) The full weight matrix evolves, so there's no rank bottleneck on the final solution. (2) Optimizer states are low-rank (r × k instead of d × k), saving proportionally more memory for large d. (3) Can be combined with 8-bit optimizers for further savings.

Disadvantages: (1) Periodic SVD computation adds overhead. (2) Cannot merge adapters (the full model changes). (3) More complex implementation.

05 Quantization-Aware Fine-Tuning

5.1 Quantization Fundamentals

Quantization maps continuous-valued parameters to a discrete set with fewer bits. The goal is to represent the model in fewer bits to save memory and speed up inference, while preserving as much model quality as possible.

Uniform (symmetric) quantization:

xq = round(x / s) where s = max(|x|) / (2b-1 - 1)
xdequant = s × xq
Quantization error: ε = x - xdequant

Asymmetric quantization (with zero-point):

xq = round(x / s + z) where s = (xmax - xmin) / (2b - 1)
xdequant = s × (xq - z)

Asymmetric is more flexible (can represent non-symmetric distributions) but adds the overhead of storing zero-points. For neural network weights, which are approximately symmetric around zero, symmetric quantization is often preferred for simplicity.

🧮 Derive It Yourself — Quantization Formulas

Quantization is just rounding with a scale factor. Build the formula from the problem statement:

1

The goal: Map a continuous value x ∈ ℝ to one of 2b discrete levels (where b is the bit-width). For 4-bit, that's 16 levels.

2

Symmetric case — find the scale: The largest representable integer in b bits (signed) is 2b-1−1. For 4-bit: 7. We want to map the range [−max|x|, +max|x|] to [−7, +7]. So: scale s = max(|x|) / (2b-1−1). Division by s maps x into [−7, +7], and rounding gives an integer: xq = round(x / s).

3

Dequantize: To recover the approximate original value: xdequant = s × xq. The error ε = x − xdequant is bounded by s/2 (half a quantization bin width).

4

Asymmetric — add a zero-point: If the values aren't centered at zero, symmetric wastes bins. Map [xmin, xmax] to [0, 2b−1]. Now: s = (xmax−xmin)/(2b−1), and z = round(−xmin/s) is the zero-point (the integer representing 0). Quantize: xq = round(x/s + z). Dequantize: xdequant = s × (xq − z).

5

NF4 twist — non-uniform bins: Instead of equally-spaced bins, place bin edges at the quantiles of N(0,1). For 16 levels, each bin captures 1/16 = 6.25% of the probability mass. Bin edges: Φ⁻¹(k/16) for k=0,...,16. Representation values are the conditional means E[X | X ∈ bink]. This minimizes E[(X − xq)²] over all 16-level quantizers when X ~ N(0,σ²).

Why NF4 is optimal — the Lloyd-Max argument: The Lloyd-Max quantizer minimizes expected distortion for a given number of levels and a known source distribution. When the source is Gaussian (as neural network weights approximately are), the optimal Lloyd-Max quantizer places bins at the Gaussian quantiles. NF4 is exactly this. If an interviewer asks "prove NF4 is optimal," invoke the Lloyd-Max optimality condition: each representation point is the centroid of its bin (conditional mean), and each bin boundary is the midpoint between adjacent representation points.
🔬 Block-Wise Quantization: Why It Matters

Quantizing an entire weight matrix with a single scale factor is suboptimal because outlier values force a wide range, reducing precision for the majority of values. Block-wise quantization partitions weights into blocks (typically 64 or 128 parameters) and uses a separate scale factor per block. This allows each block's scale to be adapted to its local range, dramatically reducing quantization error.

The trade-off is storage for the scale factors: 1 fp16 scale per 64 params = 2 bytes / 64 params = 0.25 bits/param overhead. QLoRA's double quantization reduces this to ~0.127 bits/param by quantizing the scales themselves to fp8.

Group size matters: Smaller groups = better accuracy but more overhead. 64 is the sweet spot for NF4. Some methods use group_size=128 for 4-bit (slightly worse accuracy, slightly less overhead) or group_size=32 for 2–3 bit quantization (needs finer granularity to maintain quality).

5.2 Post-Training Quantization (PTQ) — Deep Dive

GPTQ — Second-Order Quantization

Frantar et al., 2022 — GPTQ AutoGPTQ/AutoGPTQ

GPTQ quantizes each layer independently by solving an optimization problem: minimize the layer-wise output error ||WX - WqX||² subject to Wq being quantized. It uses second-order (Hessian) information to determine quantization order — parameters with high curvature (sensitive to perturbation) are quantized more carefully, and errors from quantizing one parameter are compensated by adjusting remaining parameters.

The algorithm is based on Optimal Brain Quantization (OBQ) but dramatically accelerated: GPTQ can quantize a 175B model in ~4 hours on a single GPU. Requires a calibration dataset (typically 128–512 samples) to compute the Hessian approximation. Quality at 4-bit is excellent for most models.

AWQ — Activation-Aware Weight Quantization

Lin et al., 2023 — AWQ

Key insight: not all weight channels are equally important. A small fraction of weights, corresponding to channels with large activation magnitudes, disproportionately affect output quality. AWQ identifies these "salient" channels from calibration data and protects them by scaling them before quantization (then inversely scaling the next layer's input). This redistributes the quantization budget to protect the most important weights.

AWQ is simpler and faster than GPTQ (no Hessian computation), requires less calibration data (as few as 32 samples), and produces comparable or better quality at 4-bit. Especially effective for long-context scenarios where the activation distribution shifts with position.

5.3 QAT vs. PTQ

Post-Training Quantization (PTQ)

Quantize a pre-trained model without training. Methods: GPTQ, AWQ, SqueezeLLM. Fast (minutes to hours) but quality degrades at aggressive bit-widths (≤3 bit).

When to use: Standard 4-bit inference deployment; you want fast turnaround and the quality is acceptable.

QAT (Quantization-Aware Training)

Simulate quantization during training via the Straight-Through Estimator (STE):

Forward: wq = Q(w) (quantize)
Backward: ∂L/∂w ≈ ∂L/∂wq (identity gradient)

The STE is a practical hack: quantization is a step function (non-differentiable), so during backprop we pretend it's the identity function. This is mathematically unjustified but works remarkably well in practice because the gradients still point in approximately the right direction.

When to use: Aggressive quantization (2–3 bit), or when PTQ quality isn't sufficient for your application.

🚨 Interview Nuance: The LoRA → Merge → Quantize Pipeline Problem

If you fine-tune with full-precision LoRA, merge adapters back (W' = W₀ + (α/r)BA), and then quantize the merged model to 4-bit, quality often degrades more than expected. Why? The merged weights W' have a different distribution than W₀ — the LoRA adaptation shifts the weight statistics. Quantization parameters (scale, zero-point) calibrated for W₀'s distribution are suboptimal for W'. Even recalibrating with GPTQ/AWQ helps but doesn't fully recover quality because the model never learned to operate with quantized weights.

Best practices: (1) If deploying at 4-bit, use QLoRA from the start — the adapters learn to compensate for quantization artifacts during training. (2) If you used full-precision LoRA, re-quantize the merged model using GPTQ/AWQ with calibration data from your target distribution (not generic C4 data). (3) Consider a few hundred steps of QAT-style fine-tuning after merging to adapt the model to quantization noise.

06 Instruction Tuning & Alignment

6.1 Supervised Fine-Tuning (SFT) — The Complete Picture

SFT is the first step in the alignment pipeline. It transforms a base model (which just completes text) into an assistant (which follows instructions).

LSFT(θ) = − Σ(x,y)∈D Σt=1|y| log Pθ(yt | x, y<t) [loss only on response tokens]

What SFT actually teaches the model: The base model already knows most of the information it needs. It can write code, answer questions, and generate text — but it doesn't know it should do these things in response to instructions. SFT provides examples of the desired input-output mapping, teaching the model to recognize the structure of instructions and produce appropriate responses. Think of it as teaching the model "when you see text that looks like an instruction, generate text that looks like a helpful response" rather than teaching it new knowledge.

6.2 Chat Templates — Why They're Critical

Modern instruction-tuned models use structured chat templates with special tokens. These tokens are not cosmetic — the model's behavior is deeply conditioned on them. Using the wrong template at inference can cause dramatic quality degradation because the model has never seen that token pattern during training.

6.3 Data Quality vs. Quantity — The Full Story

Zhou et al., 2023 — LIMA: Less Is More for Alignment
⚡ The LIMA Revelation

1,000 carefully curated examples can outperform 50,000+ noisy examples. The LIMA paper fine-tuned LLaMA-65B on just 1,000 hand-selected, high-quality instruction-response pairs and showed competitive or superior performance to models trained on orders of magnitude more data (like Alpaca's 52K or Vicuna's 70K). This has been consistently validated across multiple follow-up studies.

Implication: Pre-training already gives the model its knowledge and capabilities. SFT primarily teaches it the format of being an assistant. A small number of high-quality examples are sufficient to teach this format; drowning the model in noisy data teaches bad habits (verbosity, inconsistency, incorrect responses).

Quality signals to optimize for:

  • Instruction diversity: Cover a wide range of task types — factual questions, creative writing, coding, analysis, multi-step reasoning, role-play, etc. A model trained only on factual QA won't learn to write creatively, even if it could before fine-tuning.
  • Response correctness: Every response should be factually accurate and logically sound. A single wrong response in the training data can reinforce that specific error pattern, and models are remarkably good at memorizing examples.
  • Length calibration: Responses should be appropriately long for the question — short answers for simple questions, detailed answers for complex ones. If all training responses are 500+ words, the model learns to be verbose even for "What is 2+2?"
  • Formatting consistency: Pick a style and stick with it. If some examples use markdown headers and others use plain text, the model learns inconsistent formatting.
  • Refusal quality: Include examples of the model appropriately declining harmful requests. Without these, the model may either refuse too aggressively or not at all.
🚨 Decontamination: Non-Negotiable

Ensure training data doesn't contain evaluation benchmarks. If MMLU questions appear in your training data, your MMLU score is meaningless — the model has memorized the answers rather than learned the skill. This is a critical source of inflated results in the open-source community. Use n-gram overlap checks or embedding similarity to filter potential contamination. Also check for rephrased versions of benchmark questions, which can appear in synthetic data generated by strong models.

6.4 Multi-Turn Fine-Tuning

L = − Σturns Σt ∈ assistant_tokens log P(yt | full_context, y<t; θ)

Only assistant tokens contribute to the loss — user messages and system prompts are masked. This ensures the model learns to generate responses, not to generate user messages.

Sequence packing for efficiency: Short conversations waste GPU memory when padded to max sequence length. Packing concatenates multiple conversations into a single sequence, separated by special tokens, with attention masking that prevents cross-conversation information leakage. This can improve training throughput by 2–5×. However, packing requires careful implementation — each conversation must see only its own tokens in the attention mask, which means creating a block-diagonal attention matrix within each sequence.

Conversation length distribution: If your training data is mostly 1–2 turn conversations but your deployment will handle 10+ turn conversations, the model may degrade at longer contexts. Include diverse conversation lengths in training data, with emphasis on the lengths you expect in production.

07 RLHF, DPO & Preference Optimization

7.1 RLHF — Full Pipeline Explained

Ouyang et al., 2022 — InstructGPT / RLHF Ziegler et al., 2019 — Fine-Tuning LMs from Human Preferences

RLHF aligns a language model with human preferences through a three-stage pipeline. It's the technique behind ChatGPT and most major commercial models. Understanding it deeply — including its limitations — is essential.

Stage 1: SFT (Supervised Fine-Tuning)

Produces an initial policy πSFT that can follow instructions. This is the starting point for alignment — the model knows how to generate responses but doesn't yet know which responses are preferred.

Stage 2: Reward Model Training

A reward model Rφ(x, y) → ℝ is trained to predict human preferences. Human annotators are shown pairs of responses (yw, yl) to the same prompt x and select which they prefer. The reward model is trained with the Bradley-Terry loss:

LRM(φ) = −𝔼(x,yw,yl) [log σ(Rφ(x, yw) − Rφ(x, yl))]
🔬 Understanding the Bradley-Terry Model

The Bradley-Terry model assumes that the probability of preferring yw over yl is:

P(yw ≻ yl | x) = σ(R(x, yw) − R(x, yl)) = 1 / (1 + exp(−(R(x, yw) − R(x, yl))))

This is a logistic model on the difference in rewards. Key properties: (1) Only the relative reward matters, not the absolute values (adding a constant to all rewards changes nothing). (2) The model is transitive — if A is preferred to B and B is preferred to C, then A is preferred to C (since R(A) > R(B) > R(C)). (3) The gradient pushes the reward of preferred responses up and dispreferred responses down, with the push strongest when the model is most "wrong" (assigning higher reward to the dispreferred response).

Practical consideration: The reward model is typically the same architecture as the LLM (e.g., a Llama-7B) with the language modeling head replaced by a scalar head (a linear layer that outputs a single number). It's initialized from the SFT checkpoint to inherit the model's understanding of language quality. Training uses 50K–500K comparison pairs; more data helps, but quality matters more than quantity.

🧮 Derive It Yourself — Bradley-Terry Reward Model Loss

This derivation connects human preferences to a trainable loss function:

1

Assume a latent quality score: Each response y to prompt x has a "true quality" R(x,y) ∈ ℝ. Humans prefer higher-quality responses. We want to learn R from preference comparisons.

2

Model the preference probability: The probability that a human prefers yw over yl should depend on their quality difference. Use the logistic function (sigmoid): P(yw ≻ yl) = σ(R(yw) − R(yl)) = 1/(1 + exp(−(R(yw) − R(yl)))). Why sigmoid? It maps any real-valued difference to [0,1], is symmetric (swapping w and l gives 1 − P), and is the maximum-entropy model for binary outcomes given a linear sufficient statistic.

3

Write the maximum likelihood objective: Given N preference pairs, maximize Πi P(yw(i) ≻ yl(i)). Take the log: maximize Σi log σ(R(yw(i)) − R(yl(i))).

4

Negate to get a loss: L = −(1/N) Σi log σ(Rφ(x(i), yw(i)) − Rφ(x(i), yl(i))). This is the Bradley-Terry loss. It's essentially binary cross-entropy where the "logit" is the reward difference.

5

Understand the gradient intuitively: When the model is wrong (assigns higher reward to the losing response), σ is close to 0, log σ is very negative, and the gradient is large — strong push to correct the ranking. When correct, σ ≈ 1, log σ ≈ 0, gradient is small — less push. This focuses learning on the hardest pairs.

The key insight to mention: Only the difference R(yw) − R(yl) matters, not absolute values. You can add any constant to all rewards without changing the loss. This means the reward model learns a preference ordering, not absolute quality scores. This has implications for reward hacking — the model may find that a high reward doesn't correspond to high quality in absolute terms.

Stage 3: RL Optimization (PPO)

Objective: maxθ 𝔼x∼D, y∼πθ [Rφ(x, y)] − β · 𝔼x∼D [KL(πθ(·|x) || πref(·|x))]

This objective has two competing terms:

  • Reward maximization: Generate responses that score highly under the reward model. Without the KL term, the model would find degenerate solutions that exploit reward model weaknesses (e.g., producing repetitive phrases the reward model rates highly, or generating extremely long responses if the reward model has a length bias).
  • KL penalty: Stay close to the reference policy (typically πSFT). This acts as a regularizer, preventing the model from departing too far from sensible language. β controls the trade-off — higher β = more conservative (closer to SFT), lower β = more aligned with reward (but higher risk of reward hacking).
🧮 Derive It Yourself — KL Divergence for Language Models

KL divergence appears everywhere in alignment. Know how to compute it for LLMs:

1

General definition: KL(P || Q) = 𝔼x~P[log P(x) − log Q(x)] = 𝔼x~P[log(P(x)/Q(x))]. It measures how "surprised" Q would be by samples from P. Always ≥ 0 (by Jensen's inequality). Not symmetric: KL(P||Q) ≠ KL(Q||P).

2

For autoregressive LLMs: P(y|x) = Πt P(yt|x,y<t). So: KL(πθ || πref) = 𝔼y~πθ[Σt (log πθ(yt|x,y<t) − log πref(yt|x,y<t))]. This decomposes into a per-token KL sum — each token contributes log(πθ/πref) to the total.

3

In practice (for RLHF/DPO): You compute per-token log-probs from both models, subtract them, and sum. This is exactly what DPO's loss computes: the log(πθ/πref) ratios. The β multiplier converts this into the KL penalty scale. A KL of 0 means the models assign identical probabilities to all tokens; larger KL means the policy has diverged more.

Why KL(πθ || πref) and not KL(πref || πθ)? The "forward" KL (as used in RLHF) is mean-seeking: πθ tries to cover all modes of πref. The "reverse" KL would be mode-seeking: πθ would collapse to one mode. For alignment, we want the policy to remain broad and diverse (not collapse to one response style), so forward KL is correct.

PPO (Proximal Policy Optimization) optimizes this with clipped surrogate objectives:

LPPO = −min(rt(θ) · At, clip(rt(θ), 1−ε, 1+ε) · At)
where rt(θ) = πθ(at|st) / πold(at|st) is the probability ratio
At is the advantage estimate (from GAE)

In the LLM context: "states" are token sequences generated so far, "actions" are next tokens, and the "advantage" measures how much better a token choice was compared to the expected reward. The clipping (1±ε, typically ε=0.2) prevents overly large policy updates that could destabilize training.

🧮 Derive It Yourself — PPO Clipped Objective for LLMs

PPO looks intimidating but builds logically from policy gradient:

1

Start from REINFORCE (vanilla policy gradient): ∇θ J = 𝔼[Σt ∇θ log πθ(at|st) · Rt]. In LLM terms: at = next token, st = tokens so far, Rt = total reward from this point forward. Problem: high variance, unstable.

2

Replace Rt with advantage At: At = Q(st,at) − V(st) — "how much better was this action than the expected action?" Subtracting the baseline V reduces variance dramatically. In practice, use GAE (Generalized Advantage Estimation) for a smooth bias-variance trade-off.

3

Use importance sampling to reuse old data: We collected data with policy πold but are updating πθ. Correction factor: rt(θ) = πθ(at|st) / πold(at|st). The surrogate objective becomes: L = 𝔼[rt(θ) · At]. When πθ = πold, rt = 1 and we recover the original objective.

4

Add clipping to prevent large updates (→ PPO): If rt gets too far from 1 (policy changed too much), the importance sampling estimate becomes unreliable. Clip: LPPO = −𝔼[min(rt · At, clip(rt, 1−ε, 1+ε) · At)]. The min ensures we take the more pessimistic estimate — if the advantage is positive, we don't let rt grow beyond 1+ε (caps the benefit); if negative, we don't let it shrink below 1−ε (caps the penalty). This creates a "trust region" around πold.

5

For RLHF, add the KL penalty: The reward at the end of the sequence is R(x,y) − β · KL(πθ||πref). The KL penalty is typically implemented as a per-token KL: −β · (log πθ(yt|...) − log πref(yt|...)) added to the reward at each step. This keeps the policy close to the SFT model throughout training.

The progression to remember: REINFORCE → subtract baseline (advantage) → importance sampling → clip the ratio → add KL to reward. Each step addresses one practical problem: variance, data reuse, stability, and alignment drift.
⚠️ RLHF Challenges in Detail
  • Infrastructure complexity: PPO requires running 4 models simultaneously: the policy model (being trained), the reference model (for KL computation), the reward model (for scoring), and the value model (for advantage estimation). For a 7B model, that's ~56GB in bf16 just for parameters. This is why RLHF is expensive.
  • Training instability: PPO is notoriously sensitive to hyperparameters. Common failure modes: reward hacking (model finds degenerate high-reward outputs), KL collapse (model diverges completely from reference), mode collapse (model produces nearly identical outputs for all inputs). Each requires different remediation — there's no one-size-fits-all config.
  • Reward model quality ceiling: RLHF can't produce a better model than the reward model can evaluate. If the reward model can't distinguish good reasoning from plausible-sounding wrong reasoning, the aligned model will learn to produce plausible-sounding wrong reasoning.
  • Reward hacking examples: Models have learned to: produce longer outputs (many reward models have length bias), include confident-sounding phrases ("Certainly!", "Great question!"), and avoid hedging even when appropriate — all because these patterns score higher on flawed reward models.

7.2 DPO — Direct Preference Optimization

Rafailov et al., 2023 — DPO

DPO is the most important innovation in alignment since RLHF. It eliminates the reward model and RL loop entirely.

The mathematical derivation (critical for interviews):

Start from the RLHF objective. The optimal policy under the KL-constrained reward maximization has a closed-form solution:

π*(y|x) = (1/Z(x)) · πref(y|x) · exp(r(x,y) / β)

This is a Boltzmann distribution centered on the reference policy, tilted by the reward function. Taking the log and rearranging gives us an expression for the reward in terms of the policy:

r(x, y) = β · log(π*(y|x) / πref(y|x)) + β · log Z(x)

The key insight: we can express the reward as a function of the policy itself. We don't need a separate reward model — the policy is the reward model. Substituting this into the Bradley-Terry preference model:

P(yw ≻ yl) = σ(r(yw) − r(yl))
= σ(β · log(πθ(yw|x)/πref(yw|x)) − β · log(πθ(yl|x)/πref(yl|x)))

(Note: Z(x) cancels in the subtraction!) This gives us the DPO loss:

LDPO(θ) = −𝔼(x,yw,yl) [log σ(β · (log πθ(yw|x)/πref(yw|x) − log πθ(yl|x)/πref(yl|x)))]

What this loss does intuitively: It increases the log-probability of preferred responses relative to the reference model, and decreases the log-probability of rejected responses. The β parameter controls how aggressively it does this. The reference model provides the "anchor" — without it, the model would just increase the probability of preferred responses to infinity (degenerate solution).

🧮 Derive It Yourself — DPO from RLHF (The Full Derivation)

This is the most important derivation for interviews. It shows how DPO eliminates the reward model entirely. Practice until you can do it fluently on a whiteboard.

1

Start from the RLHF objective: maxπ 𝔼y~π[r(x,y)] − β · KL(π || πref). We want to maximize reward while staying close to the reference policy.

2

Expand the KL and solve for the optimum: The KL is KL(π || πref) = 𝔼y~π[log π(y|x) − log πref(y|x)]. The full objective becomes: 𝔼y~π[r(x,y) − β log π(y|x) + β log πref(y|x)]. This is a maximum-entropy / KL-constrained optimization. The closed-form solution (by calculus of variations or by recognizing the Gibbs distribution form) is: π*(y|x) = (1/Z(x)) · πref(y|x) · exp(r(x,y)/β) where Z(x) = Σy πref(y|x) · exp(r(x,y)/β) is the partition function.

3

Solve for the reward: Take log of both sides: log π*(y|x) = log πref(y|x) + r(x,y)/β − log Z(x). Rearrange: r(x,y) = β · log(π*(y|x) / πref(y|x)) + β · log Z(x). This expresses the reward in terms of the optimal policy. We don't need a separate reward model — the policy encodes the reward!

4

Substitute into Bradley-Terry: P(yw ≻ yl) = σ(r(yw) − r(yl)). The reward difference is: r(yw) − r(yl) = β[log(π*(yw|x)/πref(yw|x)) − log(π*(yl|x)/πref(yl|x))] + β·log Z(x) − β·log Z(x). The partition function Z(x) cancels! This is the magic step — we never need to compute the intractable Z(x).

5

Replace π* with πθ and write the loss: We don't have the optimal π*, but we're training πθ to approximate it. Substitute and negate the log-likelihood: LDPO(θ) = −𝔼 [log σ(β · (log(πθ(yw|x)/πref(yw|x)) − log(πθ(yl|x)/πref(yl|x))))]. This is a supervised loss — compute 4 log-probabilities (policy and reference on both preferred and rejected), combine, pass through sigmoid, take log. No RL, no sampling, no value function.

The 5 key steps to remember: (1) RLHF objective, (2) closed-form optimal policy (Gibbs/Boltzmann), (3) invert for reward, (4) substitute into Bradley-Terry (Z cancels!), (5) parameterize with πθ. Each step is one line on the whiteboard. The partition function cancellation in step 4 is the "aha moment" — emphasize it.
✅ DPO Advantages
  • 2 models (policy + reference) vs. 4 for RLHF
  • Simple supervised loop — just compute log-probs and take gradient
  • 2–4× cheaper compute than RLHF
  • No reward model overfitting — no separate model to overfit
  • Stable training — no RL instability, no reward hacking
  • Easy to implement — ~50 lines of PyTorch code for the core loss
⚠️ DPO Limitations
  • Lower ceiling than well-tuned RLHF on certain complex tasks
  • Noisy preference sensitivity — no reward model to smooth noise
  • Fixed reference model — can't do iterative alignment easily
  • No reusable reward model — can't use for rejection sampling or best-of-N
  • Offline only — can't generate new data during training (unlike online RLHF)
  • Can overfit to surface-level preference patterns (e.g., length, formatting) rather than genuine quality

7.3 Modern Preference Optimization Variants

IPO — Identity Preference Optimization

Azar et al., 2024
LIPO = 𝔼[(log(πθ(yw|x)/πref(yw|x)) − log(πθ(yl|x)/πref(yl|x)) − 1/(2β))²]

Replaces DPO's log-sigmoid with a squared loss. Why? DPO's loss can approach zero only by pushing the log-ratio to ±∞, leading to degenerate solutions where the model assigns near-zero probability to rejected responses. IPO bounds the implicit reward gap by targeting a specific margin (1/(2β)), preventing this degenerate behavior. More robust when preference data is noisy or inconsistent.

KTO — Kahneman-Tversky Optimization

Ethayarajh et al., 2024

Works with unpaired preference data — you only need a "good" or "bad" label per response, not pairwise comparisons. This is a major practical advantage because unpaired labels are much cheaper to collect (a single annotator scores each response) vs. paired comparisons (same annotator must evaluate two responses together). Inspired by Kahneman & Tversky's prospect theory, KTO applies asymmetric loss weighting: losses from undesirable outputs are weighted more heavily than gains from desirable outputs, reflecting human loss aversion.

ORPO — Odds Ratio Preference Optimization

Hong et al., 2024
LORPO = LSFT + λ · LOR (single-stage: combines SFT and preference)

Eliminates the need for a separate reference model by combining SFT loss with an odds-ratio-based preference loss. The odds ratio compares P(yw)/(1−P(yw)) vs. P(yl)/(1−P(yl)). Key benefit: Single-stage training (no SFT → DPO pipeline), simpler implementation, and no reference model forward passes during training.

SimPO — Simple Preference Optimization

Meng et al., 2024
Implicit reward: r(x, y) = (1/|y|) · Σt log πθ(yt | x, y<t) [length-normalized log-prob]

SimPO uses the model's own length-normalized log-probability as the implicit reward, completely eliminating the reference model. The length normalization is critical — without it, the model can game the reward by generating shorter (higher avg log-prob) or longer (higher total log-prob) responses depending on how the metric is computed. SimPO is often simpler and more effective than DPO, and has become a popular default for practical alignment.

GRPO — Group Relative Policy Optimization

DeepSeek-AI, 2025 — DeepSeek-R1 deepseek-ai/DeepSeek-R1
For each prompt x, sample G responses {y₁, ..., yG}
Compute rewards: ri = R(x, yi)
Normalize: Ai = (ri − mean(r₁,...,rG)) / std(r₁,...,rG)

GRPO was the key algorithm behind DeepSeek-R1's remarkable reasoning capabilities. It replaces PPO's value network with group-based advantage estimation — no separate value model needed. For each prompt, sample G responses (typically 4–16), compute rewards (can be outcome-based like math correctness), normalize within the group, and use the normalized advantage for policy gradient updates. This is simpler, more memory-efficient, and was shown to discover chain-of-thought reasoning without explicit CoT training data — purely from reward signal.

🧮 Derive It Yourself — GRPO from PPO

GRPO simplifies PPO by replacing the value network with group statistics:

1

PPO's problem: The advantage At = Q(st,at) − V(st) requires a separate value model V to estimate how good each state is. For LLMs, this is another full-size model — doubling memory requirements.

2

GRPO's insight: For each prompt x, sample G complete responses {y₁, ..., yG}. Compute the reward for each: ri = R(x, yi). The "advantage" of response i is simply how much better it was than the group average: Ai = (ri − μ) / σ, where μ = mean(r₁...rG), σ = std(r₁...rG).

3

Why does normalization work? The z-score normalization serves two purposes: (a) centering by subtracting the mean ensures responses below average get negative advantage (pushed down) and above-average get positive advantage (pushed up) — this is the baseline subtraction that reduces variance in REINFORCE. (b) dividing by std makes the advantage scale-invariant — whether rewards are in [0,1] or [0,100], the normalized advantage has unit variance.

4

Apply the PPO clipped objective with these advantages: L = −𝔼i[min(rt(θ) · Ai, clip(rt(θ), 1−ε, 1+ε) · Ai)] + β · KL(πθ||πref). Same clipping as PPO, same KL penalty — but no value model. Memory: 2 models (policy + reference) instead of PPO's 4 (policy + reference + reward + value).

Why GRPO is a big deal: It's not just simpler — the group-based advantage with verifiable rewards (e.g., "did the math answer match?") enabled DeepSeek-R1 to discover reasoning from scratch. The model learned to produce chain-of-thought because longer, more careful reasoning led to higher reward (correct answers). No CoT training data was needed. This is emergent reasoning from reward alone.

08 Mixture of Experts (MoE) Fine-Tuning

8.1 MoE Architecture

Jiang et al., 2024 — Mixtral
y = Σi ∈ TopK(G(x)) G(x)i · Ei(x) where G(x) = Softmax(Wg · x)

MoE replaces the dense FFN with N expert networks (each a full FFN) and a learned router G(x) that selects the top-k experts per token. Mixtral 8×7B has 8 experts/layer with top-2 routing: ~47B total parameters but only ~13B active per forward pass. This means MoE models have the capacity of a much larger model (good for knowledge storage) with the inference cost of a smaller one.

Why this matters for fine-tuning: The router creates a natural specialization — different experts learn to handle different types of inputs. Fine-tuning can disrupt this specialization or leverage it, depending on your approach.

🧮 Derive It Yourself — MoE Forward Pass & Load Balancing Loss

MoE formulas look complex but are straightforward once you decompose them:

1

Start with a dense FFN: y = FFN(x). One network, processes every token. Now imagine you have N different FFNs (experts). Which one should process each token?

2

Add a router: G(x) = Softmax(Wg · x) produces a probability distribution over N experts. G(x)i is the probability that expert i should handle input x.

3

Select top-k and combine: For efficiency, only activate k experts (typically k=2). y = Σi ∈ TopK(G(x)) G(x)i · Ei(x). Each selected expert's output is weighted by its gating score. This is a sparse mixture — only k out of N experts run per token.

4

Derive the load-balancing loss: Without regularization, the router might send all tokens to one expert (collapse). Define: fi = fraction of tokens routed to expert i, Pi = average gating probability for expert i. The balancing loss: Lbal = α · N · Σi fi · Pi. This is minimized when both fi and Pi are uniform (= 1/N each), giving Lbal = α. Any concentration increases the loss because the product fi·Pi is convex. The factor N normalizes so the loss equals α at perfect balance.

Why the f·P product? If we just penalized the variance of f, the router could maintain balanced routing probabilities P while actual routing (after top-k selection) is imbalanced. The product f·P captures the interaction: an expert that has both high probability AND high actual usage is penalized most, which is exactly the collapse mode we want to prevent.

8.2 Fine-Tuning Considerations — In Depth

  • Router stability: The router learns during pre-training to assign tokens to appropriate experts. Fine-tuning on a narrow domain can cause the router to collapse — routing all tokens to 1–2 experts while the rest become unused. Mitigation: Use a lower LR for router weights (e.g., 10% of the main LR), or freeze the router entirely. Monitor expert utilization: if any expert handles <5% or >50% of tokens, there's an imbalance problem.
  • Load-balancing loss: Pre-training includes an auxiliary loss that penalizes uneven expert utilization: Lbalance = α · N · Σi fi · Pi, where fi is the fraction of tokens routed to expert i and Pi is the average routing probability for expert i. Keep this loss during fine-tuning (even at a reduced weight) to prevent expert collapse.
  • LoRA for MoE — the shared vs. separate question:
    • Shared adapters: Same LoRA (B, A) applied to all experts. Fewer parameters, but ignores expert specialization — each expert gets the same modification.
    • Separate adapters: Different LoRA per expert. Preserves and extends specialization, but multiplies parameters by the number of experts. For 8 experts × 2 active, you have 8× the LoRA parameters but only use 2× per forward pass.
    • Best practice: Use separate adapters if your parameter budget allows it. The per-expert cost is small (each adapter is tiny), and the benefit of maintaining expert specialization is significant. With shared adapters, you're essentially asking all experts to adapt identically, which undermines the point of MoE.
  • Selective expert fine-tuning: Analyze which experts activate most on your domain data (run inference on a sample and log router decisions). Fine-tune only the top 2–3 most activated experts. This focuses compute where it matters and minimizes disruption to the rest of the model.

09 Continual & Domain-Adaptive Fine-Tuning

9.1 Continual Pre-Training (CPT)

When the domain shift is large enough that SFT alone can't bridge the gap (medical, legal, scientific, multilingual), you need continual pre-training — extending the next-token-prediction objective on domain-specific corpora before doing SFT:

Phase 1: θdomain = CPT(θbase, Ddomain) — next-token prediction, 1–10B tokens
Phase 2: θfinal = SFT(θdomain, Dinstructions) — instruction tuning, 10K–100K examples

Why two phases? CPT and SFT serve fundamentally different purposes. CPT teaches the model what — domain vocabulary, concepts, relationships, reasoning patterns, and style. SFT teaches the model how — to respond to instructions using its domain knowledge. Skipping CPT and going straight to SFT works for moderate domain shifts but fails when the model lacks foundational domain understanding. You can't teach a model to answer medical questions well if it doesn't understand medical terminology and reasoning.

CPT learning rate: Start at 50–80% of the original pre-training peak LR, with cosine decay. Starting too high causes catastrophic forgetting of general knowledge; starting too low makes CPT inefficient. A typical schedule for a model originally trained with peak LR 3e-4: CPT peak LR of 1.5e-4 to 2e-4, decaying over the CPT phase.

9.2 Catastrophic Forgetting — Deep Analysis

🔬 What Happens Neurally During Forgetting

Catastrophic forgetting occurs when gradient updates on new data overwrite representations that encode previous knowledge. At the neural level:

  • Pre-training distributes knowledge across many parameters via superposition — each "fact" is encoded as a pattern distributed across thousands of neurons.
  • Fine-tuning on narrow data creates strong gradients in the directions relevant to the new task, pushing weights away from the pre-trained configuration.
  • Because knowledge is distributed, modifying weights in one direction can inadvertently destroy unrelated knowledge that happened to share the same parameter subspace.
  • The severity of forgetting correlates with: (1) the magnitude of weight changes (higher LR → more forgetting), (2) the narrowness of the fine-tuning distribution (less diverse data → more forgetting), and (3) the number of parameters modified (full FT → more forgetting than LoRA).

Mitigation strategies, ranked by effectiveness:

  1. PEFT (LoRA/QLoRA): Freeze base weights entirely. Zero forgetting by construction. The adapter can be removed to restore exact pre-trained behavior. This is the default recommendation.
  2. Replay buffers (data mixing): Mix 5–20% of diverse general-purpose data into each training batch. This provides a continuous "reminder" of general capabilities. The ratio depends on the forgetting severity — start at 10% and increase if general benchmarks degrade. Use data from a similar distribution to the pre-training data (RedPajama, SlimPajama, or similar).
  3. Low learning rate: Use 1e-6 to 5e-6 for CPT/full FT (vs. 1e-5 to 5e-5 for standard SFT). Smaller weight changes = less forgetting. Combine with longer training to compensate for slower convergence.
  4. EWC (Elastic Weight Consolidation): Kirkpatrick et al., 2017 Add a penalty that discourages changes to important parameters:
    LEWC = Ltask + (λ/2) · Σi Fi · (θi − θ*i)²
    Fi is the Fisher Information for parameter i, measuring how important it is for previous tasks. Computationally expensive for large models (need to compute/store Fisher for all parameters), but effective when applicable.
    🧮 Derive It Yourself — EWC Loss

    EWC is intuitive once you see the Bayesian interpretation:

    1

    Bayesian framing: After training on task A, our knowledge of good parameters is captured by the posterior P(θ|DA). When learning task B, we want: maxθ log P(DB|θ) + log P(θ|DA). The first term is the task B loss; the second is our "prior" from task A.

    2

    Approximate the posterior: P(θ|DA) is intractable. Use a Laplace approximation (second-order Taylor around the MAP estimate θ*A): log P(θ|DA) ≈ const − ½ Σi Fi(θi − θ*A,i)², where Fi is the diagonal of the Fisher Information Matrix at θ*A.

    3

    Combine: LEWC = Ltask B(θ) + (λ/2) Σi Fi(θi − θ*A,i)². Parameters with high Fisher (important for task A) are penalized heavily for moving away from θ*A. Parameters with low Fisher (unimportant for A) are free to change.

    The Fisher intuition: Fi = 𝔼[(∂ log P(x|θ)/∂θi)²]. It measures how much the output distribution changes when you perturb θi. High Fisher = the output is sensitive to this parameter = it encodes important knowledge = protect it.
  5. Progressive LoRA: Train separate LoRA adapters for each task/domain. Compose at inference by summing or dynamically loading. Completely avoids forgetting since base weights are never modified. The trade-off is managing multiple adapters.
💡 Monitoring Forgetting in Practice

Set up a "forgetting dashboard" that tracks perplexity on 3–4 diverse held-out sets alongside your domain metrics: (1) general text (Wikipedia/C4 sample), (2) code (if applicable), (3) instruction-following (a subset of a general eval like MT-Bench), (4) your domain eval. If general perplexity increases by >5% while domain metrics improve, you're forgetting too much. If it increases >15%, you're likely losing useful capabilities.

10 Data Curation, Quality & Scaling Laws

10.1 The Data Pipeline — Detailed Walkthrough

Data quality is the single most impactful factor in fine-tuning success. A rigorous pipeline:

  1. Collection & Sourcing: Gather raw instruction-response data from multiple sources: human-written, synthetic (generated by strong models), curated from existing datasets, or extracted from domain documentation. Diversity of sources prevents monoculture bias. Track provenance for each example (source, date, quality score).
  2. Deduplication: Remove exact duplicates (hash-based) and near-duplicates (MinHash LSH is the standard algorithm — creates signature-based fingerprints and uses locality-sensitive hashing to find similar pairs in O(n) time). Duplicates cause: (1) memorization of specific examples rather than learning patterns, (2) evaluation contamination if duplicates span train/eval splits, (3) biased loss weighting (duplicated examples contribute disproportionately to gradients).
  3. Quality filtering: Use a model-based scorer — typically a reward model or quality classifier — to rate each example on a 1–5 scale. Filter out examples below threshold (typically 3/5). The Phi models (Li et al., 2023 — Textbooks Are All You Need) demonstrated that aggressive quality filtering (keeping only top-tier data) produces better models than using all available data. A 1.3B model trained on high-quality data outperformed 7B models trained on lower-quality data.
  4. Toxicity & safety filtering: Use classifiers (not keyword lists) to remove harmful content. Keyword lists have high false-positive rates (flagging medical or educational content) and miss sophisticated harmful content. Include a toxicity classifier, a PII detector, and a copyright filter.
  5. Diversity sampling: Compute embeddings for all instructions (using a sentence embedding model like all-MiniLM or BGE). Cluster with k-means or HDBSCAN. Sample uniformly across clusters, possibly with stratification by difficulty level. This ensures broad coverage — without it, your dataset might be 60% simple factual questions and 5% reasoning tasks, when you want a more balanced distribution.
  6. Length distribution matching: Plot the response length distribution and compare to your target. If all training responses are 300+ words, the model learns to be verbose. If all are under 50 words, it learns to be terse. Include a range of lengths appropriate for different complexity levels.
  7. Instruction complexity: Ensure a spectrum from simple to complex. Evol-Instruct (Xu et al., 2023 — WizardLM) synthetically increases instruction complexity through iterative rewriting: adding constraints, requiring reasoning steps, combining multiple sub-tasks. This teaches the model to handle complex, multi-faceted instructions.
  8. Final validation: Manually review 100–500 random samples from the final dataset. Look for: incorrect responses, formatting inconsistencies, instructions that are ambiguous or unanswerable, responses that don't actually address the instruction. Even with automated filtering, human review catches issues that models miss.

10.2 Scaling Laws for Fine-Tuning

Hoffmann et al., 2022 — Chinchilla Scaling Laws

Pre-training scaling laws (Chinchilla): L(N, D) ∝ N−α + D−β, with optimal compute allocation roughly N ∝ C0.5 and D ∝ C0.5. Fine-tuning dynamics differ:

  • Data efficiency is much higher: A 7B model can be effectively fine-tuned with 10K–100K examples (compared to ~1T tokens for pre-training). The model already has knowledge; you're teaching format and behavior, which requires far fewer examples.
  • Quality dominates quantity: Below ~10K examples, quality improvements have a near-linear effect on performance. Above ~50K examples, adding more data has diminishing returns unless it adds genuinely new task types or edge cases. The LIMA paper showed this starkly: 1K high-quality > 52K moderate-quality.
  • More epochs can substitute for more data: With high-quality data, training for 3–5 epochs often produces better results than 1 epoch on 3–5× more lower-quality data. However, monitor for overfitting — eval loss should guide epoch selection.
  • Model size matters for fine-tuning ceiling: A 7B model fine-tuned on perfect data will still underperform a 70B model fine-tuned on the same data for complex reasoning tasks. Fine-tuning can't add capabilities that aren't latent in the pre-trained weights.

11 Distributed Training & Infrastructure

11.1 Parallelism Strategies — When to Use What

Data Parallelism (DP)

Replicate the full model on each GPU. Each processes different data, gradients are synchronized via all-reduce (sum gradients across all GPUs, then divide by GPU count). Linear throughput scaling with GPU count. Limitation: Each GPU must hold the complete model (parameters + optimizer + gradients), so it doesn't help when a single model doesn't fit on one GPU.

FSDP / DeepSpeed ZeRO — The Workhorses of Fine-Tuning

🔬 FSDP Communication Patterns

In ZeRO-3/FSDP, parameters are sharded across GPUs. Before each layer's forward pass, an all-gather operation reconstructs the full parameters for that layer from all shards. After the forward pass, the full parameters are discarded (only the shard is kept). The backward pass does another all-gather, computes gradients, and then a reduce-scatter distributes gradient shards back. This means each layer triggers two collective communications per training step.

Practical implication: FSDP works best with high-bandwidth interconnects (NVLink at 600 GB/s within a node, InfiniBand at 200–400 Gb/s across nodes). Over PCIe (32 GB/s), the communication overhead can eat 40–60% of theoretical throughput. For fine-tuning, prefer ZeRO-2 over ZeRO-3 when the model fits in ZeRO-2 configuration, since it has less communication overhead.

Tensor Parallelism (TP)

Split individual weight matrices across GPUs. For a linear layer Y = XW where W ∈ ℝd×d:

  • Column parallel: W = [W₁ | W₂], Y = [XW₁ | XW₂]. Each GPU computes a subset of output features. Followed by all-reduce or all-gather depending on the next layer.
  • Row parallel: W = [W₁; W₂], X = [X₁ | X₂], Y = X₁W₁ + X₂W₂. Each GPU has different input features and a subset of weight rows. Followed by all-reduce.

TP requires NVLink because the communication happens multiple times per layer per step. Not practical across nodes. Typically used within a single node (2–8 GPUs) combined with FSDP across nodes.

Pipeline Parallelism (PP)

Assign different layers to different GPUs: GPU₀ = layers 0–15, GPU₁ = layers 16–31. Problem: naive PP leaves all but one GPU idle (pipeline bubble). Solutions: micro-batching (GPipe — split the batch into micro-batches that pipeline through), or interleaving (PipeDream — overlap forward and backward passes). The pipeline bubble is typically 10–30% of compute time with good scheduling.

12 Evaluation, Overfitting & Debugging

12.1 Multi-Faceted Evaluation Strategy

No single metric captures fine-tuning success. A comprehensive evaluation framework:

  • Task-specific benchmarks: MMLU (multi-domain knowledge), HumanEval/MBPP (code), GSM8K/MATH (math reasoning), ARC/HellaSwag (common sense). Always compare against: (1) the base model (to measure improvement), (2) the same model fine-tuned by others (to benchmark your pipeline), (3) the commercial frontier (to understand the gap). Use EleutherAI/lm-evaluation-harness for consistent evaluation.
  • Held-out test set from your target distribution: This is the most important metric. It should be completely unseen during training and representative of real-world usage. If your model will answer customer support queries, your eval should contain real customer support queries (not generic benchmarks). Randomly splitting your fine-tuning data is a start, but be careful of data leakage through paraphrasing or shared topics.
  • LLM-as-judge: Use GPT-4 or Claude to evaluate open-ended responses. Pairwise comparisons are more reliable than absolute scores — "Is response A or B better?" gives more consistent results than "Rate this response 1–5." Standard benchmarks: MT-Bench (multi-turn), AlpacaEval (single-turn). Caveat: LLM judges have biases — they tend to prefer longer responses, responses that use markdown formatting, and responses that start with "Certainly!" Calibrate against human judgments for your specific use case.
  • Human evaluation: Gold standard but expensive. Blind A/B testing with clear rubrics. At minimum, have domain experts manually review 50–100 outputs for correctness and quality. For alignment work, red-teaming (adversarial human evaluation) is essential.
  • Perplexity on general benchmarks: Track perplexity on C4/Wikipedia held-out data. If general perplexity increases by >10% during fine-tuning, you're experiencing significant catastrophic forgetting. This is a proxy for "is the model getting dumber in general while getting smarter on my task?"

12.2 Overfitting — Deep Diagnosis

Overfitting in fine-tuning is more dangerous than in traditional ML because the model can memorize responses verbatim while appearing to perform well on in-distribution eval.

Detection signals:

  1. Train loss ↓ but eval loss ↑: The classic indicator. Plot both curves on the same chart and identify the divergence point. Save checkpoints every 100 steps and select the best based on eval loss, not train loss.
  2. Decreased output diversity: Generate 10 responses for the same prompt with temperature 0.7. If they're nearly identical, the model has collapsed to a narrow output distribution. Measure diversity with self-BLEU or unique n-gram ratio.
  3. Verbatim memorization: Prompt the model with the first 20 tokens of a training example. If it reproduces the rest exactly, it's memorizing rather than generalizing. This is especially problematic for privacy (leaking training data at inference) and for decontamination (if benchmark answers were in training data).
  4. Sensitivity to minor prompt changes: If the model performs well on "What is X?" but poorly on "Explain X to me" or "Tell me about X", it has memorized specific prompt patterns rather than learning generalizable instruction-following.

Prevention strategies:

  • Early stopping: Stop training when eval loss starts increasing. This is the most reliable technique but requires frequent evaluation (every 50–200 steps).
  • Fewer epochs: 1–2 for large datasets. With 100K+ examples, a single epoch provides sufficient gradient signal for most tasks.
  • LoRA dropout (0.05–0.1): Apply dropout to the LoRA matrices during training. This is the dropout analog for PEFT — it regularizes the adapter by randomly zeroing out adapter outputs during training.
  • NEFTune (Noise Embedding Fine-Tuning): Jain et al., 2023 Add uniform noise to input embeddings during training: embed' = embed + α·U(-1,1). This simple trick improves generalization by 1–3% on open-ended evaluations. Works because it prevents the model from relying too precisely on the exact embedding values of training examples.
  • Data augmentation: Rephrase instructions (paraphrasing), shuffle few-shot examples, vary response styles, add/remove system prompts. This increases the effective diversity of your dataset without collecting new data.

13 Emerging Techniques (2025–2026)

13.1 Reasoning-Focused Fine-Tuning

The frontier of LLM fine-tuning in 2025–2026 is reasoning. Following DeepSeek-R1 and OpenAI o1/o3, training models to produce explicit, verifiable reasoning traces has become a central research focus.

  • Chain-of-thought SFT: Fine-tune on examples with detailed reasoning steps before the final answer. The format matters — the reasoning should be genuine problem-solving, not post-hoc explanation. Models trained on authentic reasoning chains (where the chain actually leads to the answer) outperform models trained on rationalized chains (where the answer is known and the chain is reverse-engineered).
  • Process Reward Models (PRMs): Lightman et al., 2023 Score each reasoning step independently, not just the final answer. In a 10-step chain, an outcome reward model (ORM) provides 1 bit of signal; a PRM provides 10. This enables precise credit assignment — identify exactly where reasoning went wrong. PRMs consistently outperform ORMs for best-of-N selection and for RL training signal.
  • GRPO for reasoning: DeepSeek-R1 showed that GRPO with verifiable rewards (math correctness, code tests) can discover chain-of-thought reasoning from scratch. The model was not given any CoT examples — it learned to reason step-by-step purely because doing so increased the probability of getting correct final answers. This suggests that reasoning ability is latent in pre-trained models and can be unlocked through appropriate RL incentives.
  • STaR (Self-Taught Reasoner): Zelikman et al., 2022 Iterative self-improvement: (1) generate reasoning traces for problems, (2) filter for those that reach correct answers, (3) fine-tune on the successful traces, (4) repeat. Each iteration improves reasoning quality. The model bootstraps from a few seed examples to increasingly sophisticated reasoning over multiple rounds.

13.2 RLVR — Reinforcement Learning with Verifiable Rewards

For tasks with programmatic verification (math — check answer; code — run tests; formal logic — proof checker), RLVR bypasses human annotation entirely. The reward function is binary: 1 if the output passes verification, 0 otherwise. This is: (1) perfectly accurate (no reward model errors), (2) infinitely scalable (no human bottleneck), (3) provides clear signal for RL optimization. RLVR has been shown to be remarkably effective for math (DeepSeek-R1 achieves near-frontier math scores) and code (models that pass test suites reliably).

13.3 Multimodal Fine-Tuning

Liu et al., 2023 — LLaVA InternVL2, 2024

Vision-language model (VLM) fine-tuning has distinct considerations:

  • Architecture: Most VLMs use a frozen visual encoder (SigLIP/CLIP) → projection layer → LLM. Fine-tuning options: (1) train only the projection layer (cheapest, good for format adaptation), (2) train projection + LLM (standard, best quality), (3) train everything including visual encoder (expensive, only for major domain shifts like medical imaging).
  • Resolution vs. cost: Higher image resolution → more visual tokens → quadratically more attention compute. A 384×384 image might produce 256 tokens, while 768×768 produces 1024 tokens. Dynamic resolution approaches (process images at their native resolution, up to a maximum) balance quality and efficiency.
  • Data format: Training data requires (image, question, answer) triples. For fine-tuning, generating high-quality visual QA pairs is harder than text-only — you need descriptions that accurately describe image content, spatial relationships, and visual details.

13.4 Long-Context Fine-Tuning

Chen et al., 2023 — Position Interpolation Peng et al., 2023 — YaRN

Extending context length post-training involves modifying the positional encoding:

  • Position interpolation: Scale RoPE position indices: f(θ, m) → f(θ, m × Lorig/Ltarget). This compresses positions into the originally trained range. Requires a short continued-training phase (1K–10K steps) to adapt. NTK-aware scaling modifies the frequency base instead: θ' = θ × (Ltarget/Lorig)2/(d-2), which preserves short-range position resolution better.
  • YaRN: Combines NTK scaling with attention temperature correction (divides attention logits by √(s) where s is the scale factor) for sequences beyond the interpolation range. Handles extrapolation better than pure interpolation methods.
  • Critical caveat: Extending context length makes the model accept longer inputs, but doesn't automatically make it use information from distant positions effectively. The "Lost in the Middle" phenomenon (Liu et al., 2023) shows models perform worse on information in the middle of long contexts. Continued training on long documents (100M–1B tokens with the extended context) is necessary to teach the model to attend to distant information.

13.5 Model Merging & Task Arithmetic

Ilharco et al., 2022 — Task Arithmetic Yadav et al., 2023 — TIES-Merging
  • Task vectors: ΔW = Wfinetuned − Wbase defines a direction in weight space corresponding to a task. These vectors can be:
    • Scaled: Wnew = Wbase + λΔW (λ < 1 for partial adaptation, λ > 1 for stronger effect)
    • Combined: Wmerged = Wbase + λ₁ΔW₁ + λ₂ΔW₂ (multi-task model from single-task fine-tunes)
    • Negated: Wnew = Wbase − λΔW (removes the task-specific behavior — used for unlearning)
  • TIES-Merging: Addresses interference between task vectors by: (1) trimming small-magnitude changes (noise), (2) resolving sign conflicts (when different task vectors push the same parameter in opposite directions), (3) merging only the surviving consistent changes. Significantly outperforms naive averaging.
  • SLERP (Spherical Linear Interpolation): Interpolates between two model checkpoints on the hypersphere (normalizing weight vectors before interpolation). Preserves the "direction" of each model's knowledge better than linear interpolation.
  • LoRA adapter composition: For LoRA-based fine-tunes, you can merge adapters by summing BA products: W' = W₀ + (α₁/r₁)B₁A₁ + (α₂/r₂)B₂A₂. This is particularly useful for multi-task serving where different capabilities (code, math, language-specific) are trained as separate adapters and composed based on user needs.

14 Mock Interview Questions & Model Answers

Q Explain the theoretical justification for why LoRA works. Why should weight updates be low-rank?

The theoretical foundation comes from the intrinsic dimensionality work by Aghajanyan et al. (2021), which showed that pre-trained models have very low intrinsic dimensionality for downstream tasks. Specifically, they demonstrated that 200-dimensional reparameterizations suffice for a 355M parameter model to reach 90% of full fine-tuning performance — meaning the task-specific adaptation lies in a tiny subspace of the full parameter space.

LoRA operationalizes this by constraining ΔW = BA to have rank at most r. Why is the optimal ΔW low-rank? After pre-training, the model is positioned in a well-conditioned region of the loss landscape. The downstream task requires only a small perturbation from this initialization — and this perturbation lies in a low-dimensional manifold. The singular value analysis of learned LoRA adapters confirms this: the top 1-4 singular values capture the vast majority of the adaptation signal, with remaining singular values near zero.

From an optimization perspective: the loss landscape around the pre-trained initialization is approximately quadratic (locally convex), and the Hessian has a rapidly decaying eigenspectrum. The significant eigendirections correspond to the dimensions where the loss changes meaningfully — these are exactly the directions LoRA captures with its rank-r constraint. As models get larger, they move closer to many downstream solutions during pre-training, so the required perturbation becomes proportionally smaller — explaining why PEFT methods become more effective at scale.

Q You fine-tuned with DPO and it performs well on preference eval but poorly on general benchmarks. Diagnose and fix.

This is a classic alignment tax / over-optimization scenario. I'd investigate systematically:

Diagnosis step 1 — Check β: If β is too low (e.g., 0.01), the implicit KL regularization is too weak, allowing the model to diverge far from the reference policy. The implicit reward (log π_θ/π_ref ratio) may be growing unboundedly — check training curves for this. Fix: increase β to 0.3–0.5.

Diagnosis step 2 — Analyze preference data: Is the data biased toward a narrow quality dimension? For example, if preferred responses are consistently longer and more verbose, the model learns "verbose = good" at the expense of conciseness and accuracy. Check: average length of preferred vs. rejected responses, diversity of preference signals (helpfulness, correctness, safety, conciseness), and whether the data covers reasoning tasks or only superficial quality.

Diagnosis step 3 — Reference model quality: DPO optimizes relative to the reference model. If the SFT checkpoint (reference) was weak on general benchmarks, the DPO model inherits this weakness — DPO can't add capabilities the reference model didn't have. Check: reference model's general benchmark scores.

Diagnosis step 4 — Training duration: DPO can overfit quickly, especially with small datasets. Training for 3 epochs on 10K preference pairs is very different from 1 epoch on 100K. Check: eval loss trajectory — was there an overfitting inflection point?

Fixes (in priority order): (1) Increase β. (2) Add diverse general-purpose preference pairs (20-30% of data). (3) Reduce training duration (1 epoch max). (4) Try SimPO or ORPO which may be less prone to this issue. (5) Switch to iterative DPO: use the DPO model to generate new responses, collect new preferences, and do another round with a higher β.

Q How would you fine-tune a 70B model on a cluster of 8×A100 80GB GPUs?

Option 1 — Full fine-tuning: Use FSDP (ZeRO Stage 3) + bf16 mixed precision + gradient checkpointing + Flash Attention. Memory math: 70B parameters → ~140GB in bf16 for weights alone. With ZeRO-3, parameters are sharded: ~17.5GB/GPU for params. AdamW optimizer states (2 × fp32 copies): ~560GB total, ~70GB/GPU. Gradients: ~140GB total, ~17.5GB/GPU. Per-GPU total: ~105GB. This exceeds 80GB, so gradient checkpointing is essential (reducing activation memory from ~40GB to ~12GB) and we might need to use CPU offloading for some optimizer states. Micro-batch size 1 with gradient accumulation steps 8 → effective batch size 64.

Option 2 — QLoRA (recommended unless full FT is required): Quantize to 4-bit NF4 (~35GB total, ~4.4GB/GPU with sharding). LoRA adapters in bf16 with rank 64-128 on all linear layers: ~160-320M trainable parameters. With gradient checkpointing and Flash Attention, this comfortably fits with micro-batch 2-4 per GPU. Much faster per-step and typically reaches 95%+ of full fine-tuning quality.

Framework choice: HuggingFace TRL + DeepSpeed ZeRO-3 for the most battle-tested stack, or Axolotl for config-driven reproducible pipelines. For RLHF specifically, OpenRLHF handles the multi-model scheduling well. Communication: NCCL over NVLink within the node (assuming DGX or HGX topology with NVSwitch).

Key considerations: Flash Attention is non-negotiable at this scale — it saves 40-60% activation memory on long sequences. Gradient clipping (max_norm=1.0) as a safety net against loss spikes. Checkpointing every 200 steps to enable recovery. A 2TB NVMe SSD or shared filesystem (NFS/Lustre) for checkpoints, since each full checkpoint is ~140GB in bf16.

Q What is NF4 and why is it information-theoretically optimal for neural network weights?

NF4 (4-bit NormalFloat) is a non-uniform quantization scheme designed specifically for the statistical distribution of neural network weights. The key insight: transformer weights empirically follow an approximately normal distribution N(0, σ²). Standard uniform INT4 spaces 16 bins equally across the value range — but because weights concentrate around zero (the peak of the Gaussian), most bins cover low-probability tail regions while the high-probability center is poorly resolved.

NF4 instead places bin edges at the quantiles of the standard normal distribution. This means each of the 16 bins captures exactly 1/16 of the probability mass. By the theory of optimal quantization, when each quantization bin is utilized with equal probability, the expected quantization error is minimized for a given number of bits. This is because information-theoretically, you want each code point to carry equal amounts of information — if some bins are used rarely, those bits are "wasted."

Concrete construction: Compute the 17 quantiles of N(0,1): q₀ = -∞, q₁ = Φ⁻¹(1/16), q₂ = Φ⁻¹(2/16), ..., q₁₆ = +∞. The representation values are the midpoints of these bins (the conditional mean of N(0,1) within each bin). This gives 16 non-uniformly spaced values, densely packed near zero and sparse in the tails.

Empirical impact: NF4 reduces quantization error by ~30-50% compared to uniform INT4 for typical transformer weights, which directly translates to better model quality at the same bit width. QLoRA additionally applies double quantization: the per-block scale factors (one fp32 scalar per 64 parameters) are themselves quantized to fp8, reducing the overhead from 0.5 bits/param to 0.127 bits/param — saving ~3.3GB for a 70B model.

Q How do you handle catastrophic forgetting when fine-tuning for a specialized domain?

My approach depends on the severity of the domain shift and the importance of retaining general capabilities.

For moderate shifts (e.g., customer support, specific writing style): Use LoRA/QLoRA. Since base weights are frozen, forgetting is eliminated by construction. The adapter captures domain-specific behavior while the frozen weights retain all pre-trained knowledge. This is my default recommendation because it's simple, effective, and reversible (remove the adapter to restore the base model).

For significant shifts requiring more parameter modification (e.g., new language, medical domain): A multi-pronged approach: (1) Data mixing — add 10-20% of diverse general-purpose data to each training batch. This is the single most effective technique. Use a high-quality general dataset like SlimPajama or a filtered subset of RedPajama. (2) Low learning rate with warmup — peak LR of 1e-5 to 3e-5 with 5-10% warmup. (3) Continuous monitoring — track perplexity on a held-out general benchmark (Wikipedia, C4 subset) every 100 steps. If general perplexity rises >5-10% from baseline, I'm forgetting too much and need to increase the mixing ratio or decrease LR.

For major domain shifts (e.g., English model → Japanese): Two-phase approach: (1) Continual pre-training on domain corpus (1-10B tokens) with aggressive data mixing (30-40% English/general data). (2) Domain-specific SFT with the adapted base. The CPT phase naturally overlaps with general language, reducing forgetting.

For multi-task scenarios: Progressive LoRA — train separate adapters per task, compose at inference. Zero forgetting since base weights are never modified. Can even do online adapter selection based on input classification.

Q LoRA adapter works well, but quality degrades after merging and quantizing to 4-bit. Why and how to fix?

This is a well-documented issue in the LoRA pipeline. The root cause is a distribution mismatch between the weights the quantizer was designed for and the weights it actually receives.

Detailed explanation: When you train LoRA in full precision, the base model weights W₀ are in bf16/fp32. The adapter learns BA such that W₀ + (α/r)BA produces good outputs. After merging, the new weight matrix W' = W₀ + (α/r)BA has a different statistical distribution than W₀ — the LoRA update shifts means, changes variances, and may introduce new outliers. When you then quantize W' to 4-bit, the quantization parameters (scale, zero-point per block) are computed for W's distribution but W' has a different distribution. The quantization error is therefore higher than it would be for a model that was designed to operate with quantized weights.

Additionally, during QLoRA training, the adapter implicitly learns to compensate for quantization artifacts — it sees dequantized (noisy) base weights during forward passes and adapts accordingly. Full-precision LoRA never sees this noise, so it hasn't learned to be robust to it.

Solutions in priority order: (1) Use QLoRA from the start — the adapters learn in the quantized regime, producing a system that's inherently compatible with 4-bit inference. (2) Re-quantize with target-distribution calibration — after merging, run GPTQ or AWQ using calibration data from your target distribution (not generic C4). This adapts the quantization parameters to W's actual distribution. (3) QAT-style fine-tuning after merge — do 200-500 steps of fine-tuning with simulated quantization (STE) on the merged model to adapt it to quantization noise. (4) Use higher bit-width — if 4-bit degrades too much, 6-bit or 8-bit may be acceptable for your deployment constraints.

Q Compare PRMs vs. ORMs for mathematical reasoning. When would you use each?

Outcome Reward Models (ORMs) assign a single scalar reward based on the final answer — correct or incorrect. Process Reward Models (PRMs) assign rewards to each intermediate reasoning step independently.

The credit assignment problem: In a 10-step mathematical derivation, an ORM provides 1 bit of signal: "the final answer was right/wrong." A PRM provides 10 separate evaluations: "step 1 was correct, step 2 was correct, step 3 had an error, ..." This is a 10× denser supervision signal. The model can identify exactly where reasoning went wrong — was it a calculation error in step 3, or a flawed logical step in step 7? With ORM-only feedback, the model can't distinguish between these scenarios and must average over all possible error locations.

OpenAI's work on Let's Verify Step by Step (Lightman et al., 2023) showed PRMs significantly outperform ORMs for: (1) best-of-N selection (choosing the best solution from N candidates — PRMs identify solutions with correct reasoning chains, not just lucky answers), (2) RL training signal (denser rewards lead to faster, more stable convergence).

Trade-off: PRMs require step-level annotations, which are much more expensive. Each step must be labeled as correct/incorrect by a qualified annotator. Practical compromise: Automated PRM labeling via Monte Carlo estimation. For each step in a solution, sample K completions from that point onward. The step-level "correctness score" is the fraction of completions that reach the correct final answer. This is noisy but scalable (no human annotators) and has been shown to produce effective PRMs.

When to use which: Use ORMs when you need quick iteration (much cheaper to build), for tasks where individual step quality is hard to define, or as a starting point. Use PRMs when precision matters (math competitions, formal reasoning), when you have the budget for step-level labeling, or when you can use automated labeling (verifiable tasks).

Q Design a production fine-tuning pipeline end-to-end.

Stage 1 — Data preparation (2-3 weeks): Collect 10K-50K task-specific instruction-response pairs from multiple sources (human-written, synthetic, extracted from docs). Run deduplication (MinHash LSH), quality filtering (reward model scoring, keep top 70%), safety filtering (toxicity + PII classifiers), and diversity sampling (cluster-based). Format into the target model's chat template with proper prompt masking. Split 90/5/5 into train/val/test with decontamination against all planned evaluation benchmarks (n-gram overlap check).

Stage 2 — Baseline establishment (1 day): Evaluate the base model on the test set using few-shot prompting (0, 1, 3, 5 shots). Record: accuracy/quality metrics, latency, cost per query. This is the floor your fine-tuned model must beat — and it sets expectations for what improvement is possible.

Stage 3 — Training (1-3 days): Use QLoRA (4-bit NF4 base + bf16 adapters, rank 64, all linear layers, α=128) with TRL/Axolotl. Cosine LR schedule: peak 2e-4, warmup 3%, decay to 2e-5. 1-3 epochs depending on dataset size. Eval every 200 steps on val set. Save top-3 checkpoints by val loss. Enable WandB/MLflow for experiment tracking.

Stage 4 — Evaluation (2-3 days): Multi-faceted: (1) Test set performance vs. baseline (must show clear improvement). (2) General benchmarks (MMLU, etc.) to check forgetting (must not drop >5%). (3) LLM-as-judge on 200 examples (pairwise: fine-tuned vs. base, fine-tuned vs. GPT-4). (4) Human review of 100 examples by domain experts (correctness, helpfulness, safety). (5) Red-teaming: adversarial prompts to test safety guardrails.

Stage 5 — Deployment (1-2 days): Merge LoRA adapters. Re-quantize with GPTQ/AWQ using 256 calibration samples from target distribution. Validate quantized model quality (should not drop >2% from merged model). Serve with vLLM (for GPU) or llama.cpp (for edge). Set up A/B testing in production: route 10% traffic to fine-tuned model, 90% to baseline, compare metrics over 1 week.

Throughout: Version everything — data (with hash), training configs, checkpoints, eval results. Use reproducible random seeds. Document all decisions and their rationale.

Q Explain the β parameter in DPO and its mathematical relationship to KL constraint in RLHF.

β in DPO directly corresponds to the KL penalty coefficient in the RLHF objective: maxπ 𝔼[r(x,y)] − β·KL(π||πref). The derivation shows the optimal policy is π*(y|x) ∝ πref(y|x)·exp(r(x,y)/β). Taking log and rearranging: r(x,y) = β·log(π*(y|x)/πref(y|x)) + β·log Z(x). Substituting into Bradley-Terry gives the DPO loss, where β appears as log σ(β × [log-probability ratio difference]).

What β controls intuitively: Higher β = the model stays closer to the reference policy. Each preference pair pulls the model less. The implicit reward gap between preferred and rejected responses is bounded more tightly. Lower β = each preference pair has stronger influence, the model departs more from the reference, and the implicit reward gap can grow larger.

Practical ranges: β = 0.5–0.7 → very conservative, model barely changes from SFT. Good when you want subtle alignment refinements. β = 0.1–0.3 → standard range, balanced adaptation. This is where most successful DPO runs operate. β = 0.01–0.05 → aggressive, risk of overfitting to preference data patterns.

Key nuance: Because DPO has no explicit reward model, there's no reward model to overfit (unlike RLHF where the policy can exploit reward model weaknesses). This means DPO is generally safer at lower β values than RLHF would be. However, DPO can still overfit to surface-level patterns in the preference data (length bias, formatting preferences) rather than genuine quality improvements. Monitoring output diversity and general benchmark performance is essential regardless of β choice.

Tuning strategy: Start at β = 0.1. If the model is too conservative (general benchmarks fine but preference eval barely improved), decrease to 0.05. If general benchmarks degrade, increase to 0.3. Validate on held-out preference pairs.

Q Explain the mathematical difference between DPO and RLHF. When would you choose one over the other?

RLHF: Three separate components: (1) a reward model Rφ trained on human preferences via Bradley-Terry loss, (2) a value model V for advantage estimation, (3) PPO optimization of the policy πθ to maximize 𝔼[R(x,y)] − β·KL(πθ||πref). The reward model is an explicit function that scores any response. The policy uses RL (PPO) to explore the response space, guided by the reward signal.

DPO: Derives the closed-form for the optimal policy: π*(y|x) ∝ πref(y|x)·exp(r(x,y)/β). Rearranges to express the reward as r(x,y) = β·log(πθ/πref) + C. Substitutes into Bradley-Terry to get a supervised loss directly on the policy. No separate reward model, no RL, no value function.

The mathematical distinction: RLHF separates the reward model from the policy and uses RL to bridge them. DPO collapses the reward model into the policy itself — the policy is the reward model (up to a partition function). This is mathematically elegant but has a consequence: you lose the ability to query the reward for arbitrary (x, y) pairs, which RLHF's explicit reward model allows.

Choose RLHF when: (1) You need the reward model for other purposes — rejection sampling, best-of-N selection, real-time response quality monitoring, or reward-weighted training data selection. (2) Your preference data is noisy — the reward model acts as a denoising bottleneck that smooths inconsistent annotations. (3) You want online/iterative alignment — generate new data with the current policy, score it, and continue training. (4) You're at frontier scale (like OpenAI/Anthropic) where the extra infrastructure cost is justified by marginal quality gains.

Choose DPO when: (1) You want simplicity and speed — most practical use cases. DPO achieves comparable results with 2-4× less compute and a much simpler codebase. (2) You're resource-constrained — only 2 models needed vs. 4. (3) You're doing your first alignment experiment — DPO is easier to debug and iterate on. (4) Your preference data is high-quality and consistent (less need for reward model smoothing).

Consider newer alternatives: SimPO (no reference model needed), ORPO (single-stage SFT + alignment), or GRPO (RL-style but simpler than PPO, used by DeepSeek-R1 to great effect).
